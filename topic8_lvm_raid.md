# Тема 8: Управление дисками, LVM и RAID

## 8.1. MBR vs GPT

Таблица разделов — структура в начале диска которая описывает где какой раздел начинается и заканчивается. Каждый диск имеет свою таблицу.

```
диск /dev/sda:
[таблица разделов][раздел 1][раздел 2][раздел 3]
      ↑
   MBR или GPT
```

Зачем нужна — разбить один физический диск на независимые части:
```
/dev/sda (500GB):
├── /dev/sda1 (50GB)  → / (корневая ФС)
├── /dev/sda2 (400GB) → /home
└── /dev/sda3 (50GB)  → swap
```

Кто использует:
- BIOS/UEFI — при загрузке читает таблицу, находит загрузочный раздел
- Ядро — создаёт устройства `/dev/sda1`, `/dev/sda2`...
- `fdisk`, `parted` — утилиты редактирования

```bash
fdisk -l /dev/sda      # посмотреть таблицу разделов
parted /dev/sda print  # то же через parted
```

### MBR (Master Boot Record) — старый:
```
- максимум 4 первичных раздела
- максимальный размер диска — 2TB
- загрузчик в первых 512 байтах
- BIOS
```

### GPT (GUID Partition Table) — новый:
```
- до 128 разделов
- максимальный размер — 9.4 ZB
- резервная копия таблицы в конце диска
- UEFI
- каждый раздел имеет уникальный GUID
```

Сейчас везде GPT — MBR только на старом железе.

## 8.2. PV, VG, LV в LVM

**LVM** — Logical Volume Manager. Абстракция над физическими дисками для гибкого управления пространством. По сути балансер дискового пространства — позволяет перераспределять место между томами на лету.

### Три уровня:
```
физические диски
    ↓
PV (Physical Volume) — физический том (диск/раздел инициализированный под LVM)
    ↓
VG (Volume Group) — группа томов (общий пул пространства из всех PV)
    ↓
LV (Logical Volume) — логический том (виртуальный диск нарезанный из VG)
    ↓
файловая система (ext4, xfs...)
```

Аналогия:
```
PV — кирпичи (физические диски)
VG — склад (пул из всех кирпичей)
LV — комнаты (нарезаем из склада сколько нужно)
```

### LV — главная фишка:
Для системы выглядит как обычный раздел (`/dev/mapper/vg-lv`), но физически может быть размазан по нескольким дискам внутри VG. Система не знает об этом.

```
система видит:
/dev/mapper/storage-mysql  (150GB)  ← обычный "раздел"

реально:
100GB на /dev/sda
50GB  на /dev/sdb
```

### Пример — объединить два диска:
```bash
/dev/sda (500GB) → PV
/dev/sdb (500GB) → PV
                   ↓
            VG "storage" (1TB)
                   ↓
            LV "data" (1TB) → /data
```

### Пример — место кончается:
```bash
# /var заполнилась на 95%
pvcreate /dev/sdc              # новый диск → PV
vgextend vg /dev/sdc           # добавить в VG
lvextend -L +200G /dev/vg/var  # расширить LV
resize2fs /dev/vg/var          # расширить ФС
# система не останавливалась
```

### Когда нужен LVM:
- Продакшн серверы где нужна гибкость
- Когда заранее не знаешь сколько места нужно каждому разделу
- Когда нужны снапшоты

### Когда не нужен:
- Домашний ПК с одним диском
- Простой сервер где разделы никогда не меняются

LVM не балансирует сам — это инструмент в руках администратора.

## 8.3. Расширение LV и ФС

```bash
# 1. расширить логический том:
lvextend -L +50G /dev/vg/data       # добавить 50GB
lvextend -L 150G /dev/vg/data       # установить размер 150GB
lvextend -l +100%FREE /dev/vg/data  # использовать всё свободное место в VG

# 2. расширить файловую систему:
resize2fs /dev/vg/data              # для ext4
xfs_growfs /mnt/data                # для XFS (принимает точку монтирования)

# одной командой для ext4:
lvextend -L +50G --resizefs /dev/vg/data
```

### Почему сначала LV потом ФС:
ФС не знает о LVM — думает что работает с обычным блочным устройством определённого размера. Нельзя использовать пространство которое ФС не видит.

```
до:
LV  [==100GB==]
ФС  [==100GB==]

после lvextend:
LV  [==100GB==][+50GB свободно]
ФС  [==100GB==]  ← ФС не знает о новом месте

после resize2fs:
LV  [==100GB==][+50GB свободно]
ФС  [=========150GB===========] ← перестроилась
```

`resize2fs` читает суперблок, видит что устройство выросло, перестраивает внутренние структуры.

## 8.4. Striping и mirroring — RAID 10

### Striping (чередование) — RAID 0:
Данные делятся на части и пишутся на несколько дисков параллельно:
```
файл "hello world":
диск 1: "hello "
диск 2: "world"
```
✅ Скорость чтения/записи растёт  
❌ Один диск упал → все данные потеряны

### Mirroring (зеркалирование) — RAID 1:
Одни и те же данные пишутся на несколько дисков:
```
диск 1: "hello world"
диск 2: "hello world" (копия)
```
✅ Надёжность — один диск упал, данные живы  
❌ Полезный объём вдвое меньше

### RAID 10 — striping + mirroring:
```
4 диска:
[диск1][диск2] — зеркало
[диск3][диск4] — зеркало
     ↕               ↕
     stripe (данные чередуются между парами)
```
✅ И скорость и надёжность  
✅ Можно потерять один диск из каждой пары  
❌ Полезный объём = 50% от общего

## 8.5. Статус RAID массива

```bash
cat /proc/mdstat              # статус всех RAID массивов
mdadm --detail /dev/md0       # подробный статус конкретного массива
```

```bash
cat /proc/mdstat
# md0 : active raid1 sda1[0] sdb1[1]
#       104857600 blocks [2/2] [UU]
#       [===>................] recovery = 15.2% (...)
```

`[UU]` — оба диска активны. `[U_]` — один диск выпал.

## 8.6. Swap и thrashing

Когда RAM заканчивается — ядро выгружает неактивные страницы в swap (подкачку). При обращении к ним — читает обратно (page fault → swap in). Процесс не знает что его память была в swap.

**Thrashing** — система непрерывно гоняет страницы между RAM и диском:
```bash
vmstat 1
# si — swap in (из swap в RAM)
# so — swap out (из RAM в swap)
# высокие si/so = thrashing — система почти не работает
```

## 8.7. Безопасное стирание таблицы разделов

```bash
wipefs -a /dev/sdb        # стереть все сигнатуры ФС и таблицу разделов
# или
dd if=/dev/zero of=/dev/sdb bs=1M count=10   # затереть первые 10MB нулями
```

`wipefs` — безопаснее и быстрее, стирает только сигнатуры. `dd` — физически затирает начало диска.

## 8.8. Снапшоты LVM

Снапшот — мгновенный снимок состояния LV в определённый момент времени.

Механизм **COW (Copy-On-Write)**:
```
создали снапшот → снапшот указывает на те же блоки что и оригинал
кто-то изменил блок в оригинале → старое значение копируется в снапшот → потом блок меняется
```

Снапшот не копирует все данные сразу — только изменённые блоки. Поэтому создаётся мгновенно.

```bash
lvcreate -L 10G -s -n mysnap /dev/vg/data
# -s — snapshot
# -n — имя снапшота
# 10G — место для хранения изменений (COW область)
```

Сценарий использования:
```bash
# перед рискованным обновлением БД:
lvcreate -L 10G -s -n backup /dev/vg/mysql
# обновление сломалось → откатились:
lvconvert --merge /dev/vg/backup
```

## 8.9. RAID 1 vs RAID 5

| | RAID 1 | RAID 5 |
|--|--------|--------|
| Минимум дисков | 2 | 3 |
| Отказоустойчивость | 1 диск | 1 диск |
| Полезный объём | 50% | (N-1)/N |
| Скорость чтения | Высокая | Высокая |
| Скорость записи | Средняя | Ниже (штраф на parity) |
| Механизм | Зеркало | Чередование + parity |

RAID 5 — данные + контрольная сумма (parity) распределены по всем дискам:
```
3 диска по 100GB:
диск1: данные A
диск2: данные B  
диск3: parity(A XOR B)
полезный объём: 200GB (66%)
```

Один диск упал → восстанавливается из оставшихся данных и parity.

**Когда что:**
```
RAID 1 — простота, надёжность, небольшое количество дисков
RAID 5 — экономия места, больше дисков, высокая нагрузка чтения
```

## 8.10. Снапшот LVM — когда применять

Снапшот оправдан когда:
- Перед рискованной операцией (обновление БД, миграция)
- Для консистентного бэкапа работающей системы
- Для тестирования изменений с возможностью отката

```bash
# создать:
lvcreate -L 10G -s -n snap /dev/vg/data

# смонтировать снапшот (только для чтения):
mount -o ro /dev/vg/snap /mnt/snap

# откатиться:
lvconvert --merge /dev/vg/snap

# удалить снапшот:
lvremove /dev/vg/snap
```

Снапшот растёт по мере изменений в оригинале. Если COW область заполнилась — снапшот становится невалидным. Размер выбирать с запасом.

## 8.11. Создание LV — полная последовательность

```bash
# 1. Инициализировать физический том:
pvcreate /dev/sdb1
pvdisplay              # проверить

# 2. Создать группу томов:
vgcreate myvg /dev/sdb1
vgdisplay              # проверить

# 3. Создать логический том:
lvcreate -L 50G -n mydata myvg
lvdisplay              # проверить

# 4. Создать файловую систему:
mkfs.ext4 /dev/myvg/mydata

# 5. Примонтировать:
mkdir /mnt/data
mount /dev/myvg/mydata /mnt/data

# 6. Добавить в /etc/fstab для автомонтирования:
echo "/dev/myvg/mydata /mnt/data ext4 defaults 0 2" >> /etc/fstab
```

### Полезные команды LVM:
```bash
pvs    # краткий список PV
vgs    # краткий список VG
lvs    # краткий список LV

pvdisplay   # подробно о PV
vgdisplay   # подробно о VG
lvdisplay   # подробно о LV

vgextend myvg /dev/sdc    # добавить PV в VG
vgreduce myvg /dev/sdb1   # убрать PV из VG
lvremove /dev/myvg/mydata # удалить LV
```
