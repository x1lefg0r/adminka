# Тема 1: Командная оболочка и основы работы

## 1.1. Как оболочка интерпретирует строку

Bash — это интерпретатор языка. Когда вводишь строку, он не просто "запускает команду" — он парсит её и строит **AST (дерево)**.

### Этапы обработки (слева направо внутри каждого этапа):

1. **Токенизация** — разбивка на слова и операторы
   - Пробел = разделитель токенов
   - Спецсимволы (`|`, `>`, `<`, `&`, `;`, `(`, `)`) — операторы, не часть слов
   - `|` сразу делит строку на командные сегменты

2. **Раскрытие** — переменные, подстановки команд
   - `$VAR` → значение переменной
   - `$(cmd)` → вывод команды (подстановка команды, не то же что просто запуск)
   - `~` → домашняя директория

3. **Word splitting** — повторная разбивка результатов подстановки по пробелам
   - Срабатывает на результатах `$VAR` и `$(cmd)` если они без кавычек
   - `"$VAR"` — отключает word splitting для этого значения

4. **Glob** — сопоставление шаблонов с файловой системой
   - `*` — любое количество любых символов
   - `?` — ровно один символ
   - `**` — рекурсивно (только если включён `globstar`)
   - Оболочка идёт в ФС, ищет совпадения, подставляет список файлов
   - Если совпадений нет — передаёт шаблон буквально (баг-ловушка)

5. **Запуск**

### Пример с pipe:
```bash
ls | grep foo
```
- Токенизация: видит `|`, делит на два сегмента `[ls]` и `[grep, foo]`
- Строит дерево:
  ```
  pipeline
  ├── command: ls
  └── command: grep
          └── arg: foo
  ```
- При запуске: создаёт pipe (буфер в ядре), форкает два процесса, подменяет им файловые дескрипторы

### Экранирование:
```bash
echo \*        # выведет *, не список файлов
echo '$HOME'   # выведет $HOME буквально
echo "\$HOME"  # то же через двойные кавычки
```

## 1.2. Переменные окружения vs локальные переменные

```bash
LOCAL=hello          # локальная — только в текущей оболочке, не передаётся дочерним
export ENV=hello     # переменная окружения — передаётся дочерним процессам
```

### Ключевые моменты:
- `export` — не создаёт переменную, а **помечает** её флагом "передавать дочерним"
- Внутри bash хранится как `{ value: "hello", exported: true }`
- Дочерний получает **копию** — изменения не влияют на родителя
- Наследование только **вниз**, никогда вверх
- Соседние скрипты не знают друг о друге — у каждого своя копия от родителя
- `export -n VAR` — снять флаг экспорта

```bash
# parent.sh
export VAR=hello
./child.sh       # child меняет VAR=world внутри себя
echo $VAR        # → hello, родитель не знает об изменении
```

## 1.3. Алгоритм поиска исполняемого файла

Порядок поиска при вводе команды:

1. **Алиасы** — есть ли `alias ls=...`?
2. **Функции** — есть ли функция с таким именем?
3. **Builtins** — встроенные команды bash (`cd`, `echo`, `export` и др.) — не файлы на диске
4. **PATH** — по директориям слева направо, ищет файл с битом исполнения `x`

```bash
which ls        # где нашёл в PATH
type ls         # что именно — alias/function/builtin/file
type cd         # → cd is a shell builtin
```

### PATH — как работает:
```bash
echo $PATH
# → /usr/local/bin:/usr/bin:/bin:/usr/sbin
```
- Строка режется по `:` (захардкожено в bash)
- Для каждой директории проверяет: существует ли файл с таким именем и битом `x`?
- Нашёл — запускает, дальше не идёт
- Поиск **не рекурсивный** — только в самой директории, не в подкаталогах

### Если в имени есть `/`:
```bash
/bin/ls          # абсолютный путь — PATH не смотрит вообще
./myscript.sh    # то же самое
```

### Принудительно обойти алиасы/функции:
```bash
\ls              # экранирование отключает alias
command ls       # искать только в PATH
builtin cd       # только builtin
```

### Команды в PATH — что за файлы:
```bash
file /usr/bin/ls   # → ELF 64-bit executable (скомпилированный бинарник)
```
Либо бинарник (ELF), либо скрипт с shebang (`#!/bin/bash`). Главное — бит `x`.

## 1.4. Aliases vs функции

### Alias — простая замена строки:
```bash
alias ll='ls -la'
alias gs='git status'
```
- Раскрывается **до токенизации** — раньше всего остального
- Живёт в памяти сессии, прописывают в `~/.bashrc`
- Аргументы прилепляются автоматически в конец
- **Не передаётся** дочерним процессам даже через `export`

### Функция — полноценный код:
```bash
mkcd() {
    mkdir -p "$1"
    cd "$1"
}
```
- Живёт в памяти сессии
- Можно передать дочерним: `export -f mkcd`
- Имеет свой контекст: `$1`, `$2`, локальные переменные

### Когда функция строго обязательна:
```bash
# 1. Аргументы в произвольном месте (алиас не может)
backup() { cp "$1" "$1.bak"; }

# 2. Несколько команд с логикой
deploy() { cd /app && git pull && npm install; }

# 3. Условия, циклы
retry() {
    for i in 1 2 3; do
        "$@" && return 0
        sleep 1
    done
}

# 4. Локальные переменные
calc() {
    local result=$(( $1 + $2 ))
    echo $result
}
```

### Почему не переменная вместо алиаса:
```bash
VAR='ls -la'
$VAR       # иногда работает через word splitting, но костыль
"$VAR"     # не сработает — ищет команду "ls -la" целиком
```
Переменная хранит данные, алиас — замена команды.

## 1.5. Список всех функций оболочки

```bash
declare -f   # все функции с телом
declare -F   # только имена функций
```

Показывает функции текущей сессии. В дочернем процессе — только экспортированные через `export -f`.

## 1.6. $HOME и текущий каталог в одной строке

```bash
echo "$HOME $(pwd)"
# или без дочернего процесса:
echo "$HOME $PWD"
```

### Способы получить домашнюю директорию:
```bash
echo $HOME          # переменная окружения
echo ~              # glob раскрытие
echo ~username      # домашняя директория конкретного пользователя
echo ~root          # → /root
getent passwd $USER | cut -d: -f6  # напрямую из /etc/passwd
```

`$HOME` берётся из `/etc/passwd` при логине. `$PWD` — переменная которую bash обновляет при каждом `cd`.

## 1.7. History expansion

```bash
!!     # вся предыдущая команда
!^     # первый аргумент === !:1
!$     # последний аргумент === !:-1
!*     # все аргументы === !:1-$
```

### Синтаксис токенов (`!:N`):
```bash
rm -rf /tmp/test
# токен 0: rm
# токен 1: -rf
# токен 2: /tmp/test

!:0    # → rm
!:1    # → -rf
!:2    # → /tmp/test
!:1-2  # → -rf /tmp/test (диапазон)
!:0-$  # === !! (вся команда)
```

`^` = начало, `$` = конец (как в regex).

### Проверить не запуская:
```bash
echo !!   # покажет что раскроется
```

### Опасность:
- В интерактивном терминале можно случайно повторить деструктивную команду
- В скриптах history expansion **выключен по умолчанию**
- Опасно в паре с `eval`

### Другие примеры:
```bash
sudo !!          # повторить предыдущую команду с sudo
cd !$            # перейти в последний аргумент предыдущей команды
!:0-$            # вся команда целиком
```

## 1.8. PATH и временное добавление каталога

```bash
export PATH="/my/dir:$PATH"   # в начало — приоритет выше
export PATH="$PATH:/my/dir"   # в конец — приоритет ниже
```

`$PATH` раскрывается сразу в текущее значение — никакой рекурсии, просто конкатенация строк.

**Временно** = только текущая сессия. Постоянно = в `~/.bashrc`.

### На один запуск:
```bash
PATH="/my/dir:$PATH" ./script.sh   # только для этой команды
```

### Никогда не добавляй `.` в PATH:
```bash
export PATH=".:$PATH"   # опасно — файл ls в текущей директории выполнится вместо системного
```

## 1.9. Одинарные vs двойные кавычки

### Двойные кавычки — раскрывается:
- `$VAR` — переменные
- `$(cmd)` — подстановка команд
- `\\` — экранирование

### Двойные кавычки — НЕ раскрывается:
- Глобы `*`, `?`
- Word splitting отключён для содержимого

### Одинарные кавычки — абсолютно всё буквально:
```bash
VAR=world
echo "$VAR"    # → world
echo '$VAR'    # → $VAR
echo "$(date)" # → Mon Jun...
echo '$(date)' # → $(date)
echo '\n'      # → \n буквально
```

**Правило:** всегда бери переменные в двойные кавычки `"$VAR"` — защита от word splitting и glob.

## 1.10. Поиск в истории без интерактивного поиска

```bash
history | grep "нужная строка"
grep "нужная строка" ~/.bash_history
```

### `history` vs `~/.bash_history`:
- `history` — история текущей сессии в памяти (включая незаписанные команды)
- `~/.bash_history` — файл на диске, пишется **при закрытии сессии**
- Если терминал упал — текущая сессия в файл не запишется

### Выполнить по номеру:
```bash
history | grep "git"
# → 342  git commit -m "fix"
!342   # выполнит команду номер 342
```

## 1.11. Все переменные окружения по алфавиту

```bash
env | sort        # только переменные окружения
printenv | sort   # то же самое
set | sort        # всё: окружение + локальные + функции (намного больше)
```

### `env` умеет запускать команды с изменённым окружением:
```bash
env VAR=hello ./script.sh        # прокинуть переменную на один запуск
env -i PATH=/bin ./script.sh     # запустить с чистым окружением
```
Текущее окружение не трогается — VAR появляется только внутри скрипта.

## 1.12. Ctrl+C, Ctrl+D, Ctrl+Z

| Комбинация | Сигнал | Что происходит |
|------------|--------|----------------|
| `Ctrl+C` | SIGINT | Прерывает процесс. Процесс **может поймать и обработать** |
| `Ctrl+D` | — | Не сигнал. Закрывает stdin (EOF). Программа получает "данных больше нет" и завершается |
| `Ctrl+Z` | SIGTSTP | Приостанавливает процесс, отправляет в background. Процесс не умирает |

### После Ctrl+Z:
```bash
jobs        # список приостановленных процессов
fg          # вернуть на передний план
bg          # продолжить в фоне
kill %1     # убить первый job
```

### Ctrl+D подробнее:
Программы читают stdin в цикле. EOF = "ввод закончился" → программа завершает цикл. Поэтому `cat`, `python3`, сам bash — все завершаются на Ctrl+D.

### Если процесс игнорирует Ctrl+C:
```bash
trap '' SIGINT   # скрипт игнорирует SIGINT
```
Тогда используй `Ctrl+\` (SIGQUIT) — убивает и создаёт **core dump** (снимок памяти процесса на диске для отладки).

### trap:
```bash
trap 'команда' СИГНАЛ              # выполнить при сигнале
trap '' SIGINT                     # игнорировать
trap - SIGINT                      # сбросить в дефолт
trap 'rm -f /tmp/lock; exit' SIGINT SIGTERM  # почистить перед выходом
```

### eval (опасная штука):
```bash
eval "$cmd"   # выполняет строку как команду, прогоняет через полный парсер bash
```
Нужен когда строка содержит операторы (`|`, `;`). Опасен с пользовательским вводом:
```bash
user_input="foo; rm -rf /"
eval "echo $user_input"   # выполнит rm -rf /
```
**Правило:** никогда не используй `eval` с пользовательским вводом.
