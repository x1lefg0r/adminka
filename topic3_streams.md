# Тема 3: Текстовые потоки, перенаправления и конвейеры

## Шпаргалка по перенаправлениям

```
>    # stdout → файл (перезапись)
>>   # stdout → файл (добавить в конец)
<    # файл → stdin
<<<  # строка → stdin (here-string)
<<   # here-document → stdin (многострочно)
2>   # stderr → файл
2>>  # stderr → файл (добавить)
2>&1 # stderr → туда же куда stdout
&>   # stdout + stderr → файл (перезапись)
&>>  # stdout + stderr → файл (добавить)
```

Логика стрелок: куда смотрит стрелка — туда идёт поток.

## 3.1. Файловые дескрипторы 1 и 2

- `1` — stdout, стандартный вывод
- `2` — stderr, стандартный поток ошибок

Почему важно разделять в скриптах:
```bash
ls /существующий /несуществующий
# stdout: /существующий/file.txt
# stderr: ls: /несуществующий: No such file or directory
```

Если не разделять — ошибки мешаются с данными:
```bash
ls /path | grep "something"          # grep получит и вывод и ошибки вперемешку
ls /path 2>/dev/null | grep "something"  # правильно — ошибки выбросить
```

### `/dev/null` — чёрная дыра:
Всё что туда пишешь — исчезает. Читать из него — сразу EOF.

Антипаттерн — заглушать всё подряд:
```bash
command 2>/dev/null   # если команда упала — никогда не узнаешь почему
```

## 3.2. Добавить stdout и stderr в один лог-файл

```bash
command >> log.txt 2>&1   # правильно
command 2>&1 >> log.txt   # НЕПРАВИЛЬНО — stderr остаётся на терминале
```

Порядок важен: сначала stdout в файл, потом stderr туда же куда смотрит stdout.

```bash
command &>> log.txt   # синтаксический сахар, то же самое
```

### `2>&1` — идея:
`>&` означает "перенаправить дескриптор в другой дескриптор":
```bash
2>&1   # stderr → туда же куда stdout
1>&2   # stdout → туда же куда stderr
```

`&` — disambiguator, говорит bash что `1` это номер дескриптора, не имя файла:
```bash
2>1    # перенаправить stderr в файл с именем "1"
2>&1   # перенаправить stderr в дескриптор 1
```

## 3.3. Как конвейер (pipe) обрабатывает поток данных

```bash
ls | grep foo
```

Ядро создаёт pipe — буфер в памяти (обычно 64KB) с двумя концами:
- пишущий конец → stdout `ls`
- читающий конец → stdin `grep`

Оба процесса запускаются **одновременно**, не последовательно.

### Если принимающий процесс завершается раньше:
```bash
ls -la | head -5   # head читает 5 строк и выходит
```
`ls` продолжает писать → pipe заполнен → `ls` получает **SIGPIPE** → завершается. Нормальное поведение, не ошибка. Именно так `head`, `less`, `grep` останавливают бесконечные потоки.

## 3.4. Here-string и Here-document

### Here-string (`<<<`):
```bash
grep "foo" <<< "$VAR"   # передать переменную как stdin
```

Без лишнего процесса в отличие от `echo "$VAR" | grep "foo"`.

```bash
# read из строки:
read a b c <<< "one two three"
echo $a   # → one

# калькулятор:
bc <<< "2 + 2"   # → 4
```

### Here-document (`<<`):
```bash
cat << EOF
строка 1
строка 2
EOF
```

`EOF` — просто метка, можно любое слово. Закрывающая метка — одна на строке без пробелов.

```bash
cat << КОНЕЦ
строка
КОНЕЦ

# с табами (<<-) — игнорирует ведущие табы:
cat <<- EOF
	строка   # таб игнорируется
	EOF      # таб перед меткой тоже игнорируется
```

Реальный сценарий:
```bash
ssh user@host << EOF
cd /var/log
tail -100 nginx.log
EOF
```

## 3.5. Pipe vs Process substitution

Pipe — один поток. `diff` нужно два входа — pipe не справится.

```bash
diff <(ls /dir1) <(ls /dir2)   # process substitution
```

`<(cmd)` — запускает команду, вывод делает доступным как временный файл через `/dev/fd/N`. Команда думает что получила файл, на самом деле поток.

```bash
# pipe — один поток, последовательно:
cmd1 | cmd2

# process substitution — несколько потоков одновременно:
diff <(cmd1) <(cmd2)
```

## 3.6. Вывод и на терминал и в файл — tee

```bash
command | tee file.txt      # перезапись
command | tee -a file.txt   # append
command | tee file1.txt file2.txt   # в несколько файлов
```

`tee` — читает stdin, пишет одновременно в stdout и файл. Как буква T.

```bash
cat access.log | tee raw.log | grep "ERROR" | tee errors.log
# сохранили сырой лог → отфильтровали → сохранили ошибки
```

## 3.7. Именованный канал (FIFO)

Обычный pipe — анонимный, только между родственными процессами (форк).

FIFO — pipe как файл в файловой системе, между любыми несвязанными процессами:

```bash
mkfifo /tmp/mypipe        # создать
ls -la /tmp/mypipe        # → prw-r--r-- (p = pipe)
```

```bash
# терминал 1:
cat /tmp/mypipe           # ждёт данных, блокируется

# терминал 2:
echo "hello" > /tmp/mypipe   # → терминал 1 вывел "hello"
```

Сценарий: два независимых процесса обмениваются данными через FIFO без знания друг о друге.

## 3.8. Отбросить stderr, stdout в пейджер

```bash
command 2>/dev/null | less
```

### Пейджер (`less`):
Программа для постраничного просмотра вывода.

```bash
cat /var/log/syslog | less   # останавливается, можно скроллить
```

Управление:
```
пробел / f   — страница вниз
b            — страница вверх
/pattern     — поиск вперёд
?pattern     — поиск назад
n            — следующее совпадение
q            — выйти
G            — конец файла
g            — начало файла
```

`less` > `more` — `more` только вперёд, `less` в обе стороны. Название — каламбур ("less is more").

## 3.9. Фильтр с выводом на экран и в файл

```bash
ls | grep "conf" | tee filtered.txt
```

### grep:
```bash
grep "conf"        # case-sensitive поиск подстроки
grep -i "conf"     # ignore case
grep -v "conf"     # invert — строки которые НЕ содержат
grep -n "conf"     # number — показать номера строк
grep -r "conf" /etc   # recursive — по директории
grep -E "foo|bar"     # extended regex
```
