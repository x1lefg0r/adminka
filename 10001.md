# Тема 17: Мониторинг производительности и аудит

## 17.1. Load Average — средняя нагрузка

```bash
uptime
# 15:00:00 up 5 days, load average: 1.50, 0.75, 0.50
#                                    ↑     ↑     ↑
#                                   1мин  5мин  15мин
```

Load average — среднее количество процессов в очереди выполнения + ожидающих I/O за период.

### Как интерпретировать:
```
на 1-ядерной системе:
LA = 1.0 → ядро загружено на 100%, очереди нет
LA = 2.0 → ядро загружено на 100%, ещё столько же ждут
LA = 0.5 → ядро загружено на 50%

на 4-ядерной системе:
LA = 4.0 → все ядра загружены на 100%
LA = 2.0 → 50% загрузка
LA = 8.0 → система перегружена вдвое
```

```bash
nproc          # количество ядер CPU
lscpu          # подробная информация о CPU
```

Тренд важнее моментального значения:
```
LA 1мин > LA 15мин → нагрузка растёт
LA 1мин < LA 15мин → нагрузка спадает
```

## 17.2. Buffers vs Cache в памяти

```bash
free -h
#               total   used    free    shared  buff/cache  available
# Mem:          16G     4G      8G      100M    4G          11G
```

### Buffers:
```
буферы для блочных устройств (диск)
временное хранение данных при записи на диск
ядро накапливает записи и пишет блоками (эффективнее)
```

### Cache (page cache):
```
кэш содержимого файлов
файл прочитали → ядро держит в памяти
следующее чтение → из памяти (быстро)
```

**Важно:** `buff/cache` — не занятая память. Ядро освобождает её при нехватке памяти процессам. `available` — реально доступная память.

```bash
# освободить кaches (обычно не нужно — ядро сделает само):
echo 3 > /proc/sys/vm/drop_caches
```

## 17.4. iowait в метриках CPU

```bash
top
# %Cpu(s): 5.0 us, 2.0 sy, 0.0 ni, 83.0 id, 10.0 wa, 0.0 hi, 0.0 si
#                                              ↑
#                                           iowait
```

**iowait** — процент времени когда CPU был свободен но процессы ждали завершения I/O операций.

### Почему высокий iowait не всегда = проблема диска:
```
iowait означает: CPU мог бы работать, но ждал I/O
причины:
- медленный диск (реальная проблема)
- нормальное поведение (много операций чтения/записи)
- сеть (NFS, сетевые диски)
- swap thrashing

диагностика: смотреть вместе с iostat
высокий iowait + высокий await в iostat = проблема диска
высокий iowait + низкий await = много операций, диск справляется
```

## 17.5. Открытые файлы и сокеты процесса

```bash
lsof -p 1234        # все открытые файлы процесса с PID 1234
lsof -p 1234 -i     # только сетевые соединения процесса
ls /proc/1234/fd/   # то же через /proc
```

Уже знаем из темы 2 и 5. Здесь акцент на диагностику:
```bash
# процесс висит — смотрим что он держит:
lsof -p 1234
# видим: много открытых соединений → утечка соединений
# видим: файл (deleted) → не освобождает место

# сколько файлов открыто:
ls /proc/1234/fd/ | wc -l
```

## 17.6. Steal time в виртуализированной среде

```bash
top
# %Cpu(s): 5.0 us, 2.0 sy, 0.0 ni, 83.0 id, 0.0 wa, 0.0 hi, 0.0 si, 5.0 st
#                                                                         ↑
#                                                                       steal
```

**Steal time (st)** — время которое гипервизор "украл" у твоей VM для других VM на том же физическом хосте.

```
физический сервер: 8 ядер
VM1: 4 виртуальных ядра
VM2: 4 виртуальных ядра
VM3: 4 виртуальных ядра (overcommit)

гипервизор отдаёт ядра тому кто активен сейчас
VM1 хочет CPU → гипервизор занят VM2 → steal time у VM1
```

Высокий steal (>5-10%) = твой хостинг-провайдер перегрузил физический сервер. Нужно жаловаться или мигрировать.

## 17.7. Трассировка сетевых системных вызовов

```bash
strace -p 1234 -e trace=network
# -p = attach к живому процессу
# -e trace=network = только сетевые syscall-ы (connect, accept, send, recv)
```

Для диагностики таймаутов соединения:
```bash
strace -p 1234 -e trace=connect,sendto,recvfrom -T
# -T = показать время выполнения каждого syscall

# видим:
# connect(3, {AF_INET, 10.0.0.1, 5432}, ...) = -1 ETIMEDOUT (110.5s)
# → соединение к 10.0.0.1:5432 таймаутится через 110 секунд
```

## 17.8. Потери пакетов на уровне интерфейса

```bash
ip -s link show eth0
# RX: bytes  packets  errors  dropped  overrun  mcast
#     ...    1000000  0       150      0        0
#                             ↑
#                           dropped — потери пакетов
```

### Причины потерь — кольцевые буферы (ring buffers):
```
сетевая карта принимает пакеты в кольцевой буфер (ring buffer)
если ядро не успевает читать → буфер переполняется → пакеты теряются
```

```bash
ethtool -S eth0 | grep -i drop   # статистика карты
ethtool -g eth0                   # размер ring buffer
ethtool -G eth0 rx 4096           # увеличить ring buffer
```

```bash
# смотреть статистику сети в реальном времени:
watch -n1 "ip -s link show eth0"
sar -n DEV 1    # статистика сетевых интерфейсов
```

## 17.9. Ключевые метрики производительности

### CPU:
```bash
top / htop     # общая картина
mpstat -P ALL  # по каждому ядру отдельно
sar -u 1 10    # CPU каждую секунду 10 раз
```

```
us (user)   — пользовательские процессы
sy (system) — ядро
wa (iowait) — ожидание I/O
st (steal)  — украдено гипервизором
id (idle)   — простой
```

### Память:
```bash
free -h        # общая картина
vmstat 1       # динамика памяти и swap
sar -r 1       # история использования памяти
```

### I/O диска:
```bash
iostat -x 1    # по устройствам
iotop          # по процессам
```

### Сеть:
```bash
iftop          # трафик по интерфейсам в реальном времени
nethogs        # трафик по процессам
ss -tnp        # соединения
```

## 17.10. TCP соединения ESTABLISHED с процессами

```bash
ss -tnp state established
# или
netstat -tnp | grep ESTABLISHED
```

```bash
ss -tnp state established
# Recv-Q Send-Q  Local Address:Port  Peer Address:Port  Process
```

## 17.11. Диагностика утечки памяти (*) — USE методология

**USE методология** (Brendan Gregg) — для каждого ресурса проверить:
```
U — Utilization  (утилизация) — насколько занят ресурс
S — Saturation   (насыщение)  — есть ли очередь/ожидание
E — Errors       (ошибки)     — есть ли ошибки
```

Применение к сценарию "высокий LA, низкий CPU":
```bash
# CPU:
vmstat 1
# смотрим: r (run queue) > кол-во ядер → насыщение CPU
# смотрим: b (blocked) > 0 → процессы ждут I/O

# Диск:
iostat -x 1
# смотрим: await > 10ms → диск медленный
# смотрим: util > 80% → диск насыщен

# Память:
vmstat 1
# смотрим: si/so > 0 → swap активен → насыщение памяти
free -h
# смотрим: available < 10% → память почти закончилась

# Итог:
# высокий LA + низкий CPU + высокий iowait + swap = нехватка памяти → thrashing
```

### Диагностика скрытых потребителей памяти:
```bash
# slab allocator — память ядра:
cat /proc/slabinfo | sort -k3 -rn | head -20
slabtop   # интерактивно

# page cache:
cat /proc/meminfo | grep -E "Cached|Buffers|Slab"

# сетевые буферы сокетов:
ss -m   # память сокетов
cat /proc/net/sockstat
```

## Быстрая диагностика — с чего начать

```bash
# 1. Общая картина:
uptime                 # LA
free -h                # память
df -h                  # диск

# 2. Что грузит CPU:
top -o %CPU            # сортировать по CPU
ps aux --sort=-%cpu | head -10

# 3. Что грузит память:
top -o %MEM
ps aux --sort=-%mem | head -10

# 4. Что грузит диск:
iotop -o

# 5. Что происходит в сети:
ss -tnp state established | wc -l   # количество соединений
iftop                                # трафик

# 6. Последние ошибки:
journalctl -p err -b --since "1 hour ago"
dmesg | tail -50   # сообщения ядра
```

## Полезные утилиты

```bash
# процессы:
top / htop    # интерактивный мониторинг
ps aux        # снимок всех процессов
pstree        # дерево процессов

# CPU:
mpstat        # по ядрам
sar           # история (нужен sysstat)

# память:
free          # краткая сводка
vmstat        # динамика
/proc/meminfo # подробно

# диск:
df            # использование ФС
du            # использование директорий
iostat        # I/O статистика устройств
iotop         # I/O по процессам

# сеть:
ss            # сокеты и соединения
ip            # интерфейсы и маршрутизация
iftop         # трафик по интерфейсам
nethogs       # трафик по процессам
tcpdump       # захват пакетов

# универсальные:
lsof          # открытые файлы и соединения
strace        # системные вызовы процесса
dstat         # всё в одном (CPU, диск, сеть, память)
```
