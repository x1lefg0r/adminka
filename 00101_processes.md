# Тема 5: Процессы, потоки, приоритеты и фоновое выполнение

## Сигналы — шпаргалка

```
Ctrl+C  → SIGINT  (2)  — мягкое прерывание, можно поймать
Ctrl+\  → SIGQUIT (3)  — жёстче + core dump, можно поймать
Ctrl+Z  → SIGTSTP (20) — приостановить, можно поймать

SIGTERM (15) — мягкое завершение, можно поймать (kill по умолчанию)
SIGHUP  (1)  — перечитать конфиг (для демонов)
SIGKILL (9)  — принудительное, НЕЛЬЗЯ поймать или игнорировать
```

`kill` — не только убивает, это утилита отправки любого сигнала по PID:
```bash
kill 1234          # === kill -SIGTERM 1234
kill -9 1234       # === kill -SIGKILL 1234
kill -0 1234       # проверить существует ли процесс (не посылает сигнал)
```

## 5.1. Uninterruptible sleep (состояние D)

```bash
ps aux
# STAT = D   ← uninterruptible sleep
```

Процесс ждёт завершения операции ввода-вывода и **не может быть прерван сигналами** — даже SIGKILL не поможет. Ядро держит его пока операция не завершится.

Типичные причины:
- Ожидание диска (медленный NFS, зависший диск)
- Ожидание ядерного ресурса
- Баг в драйвере

Единственный способ избавиться — дождаться завершения операции или перезагрузить систему.

## 5.2. Список процессов пользователя по памяти

```bash
ps -u username --sort=-%mem
```

### ps — process status:
```bash
ps aux
# a — all users (все пользователи)
# u — user-oriented format (владелец, CPU, память)
# x — include processes without terminal (фоновые демоны)
```

`--sort=-%mem` — сортировка по памяти по убыванию (`-` = убывание).

### Найти PID:
```bash
pidof nginx          # PID по точному имени
pgrep nginx          # по подстроке, умеет regex
pgrep -f "python script.py"  # -f = по полной командной строке
ps aux | grep [n]ginx  # хак — grep не найдёт сам себя
```

## 5.3. nice vs приоритеты реального времени

Ядро не может запускать все процессы одновременно — планировщик решает кому дать CPU прямо сейчас. Приоритеты говорят планировщику кто важнее.

### nice — обычный приоритет:
```bash
# значения от -20 (высший) до +19 (низший)
nice -n 19 tar -czf backup.tar.gz /data   # низкий приоритет — не мешает серверу
nice -n -20 ./critical                     # высокий приоритет — только root
renice -n 5 -p 1234                        # изменить приоритет живого процесса
```

Логика названия: чем "nice" (вежливее) процесс — тем больше уступает, тем выше число.

### RT приоритеты — выше всех nice:
```bash
chrt -f 50 ./audio_daemon   # SCHED_FIFO, приоритет 1-99
chrt -r 50 ./critical       # SCHED_RR (round-robin)
```

Используют для критичных к задержкам задач (аудио, промышленные системы).

## 5.4. Корректное завершение с таймаутом

```bash
kill -SIGTERM 1234
sleep 10
kill -0 1234 2>/dev/null && kill -SIGKILL 1234
```

Почему не сразу SIGKILL: процесс не успеет закрыть файлы, соединения с БД, удалить lock-файлы → повреждённые данные.

## 5.5. Процесс-зомби

Когда процесс завершается — ядро не удаляет его сразу. Оставляет запись со статусом `Z` пока родитель не вызовет `wait()` и не заберёт код возврата.

```bash
ps aux | grep Z   # найти зомби
```

Аналогия: сотрудник вернулся с отчётом и стоит в дверях. Пока не примешь отчёт — не уйдёт. Стоит в дверях = зомби.

- Зомби не потребляют CPU/память — только запись в таблице
- Нельзя убить через `kill -9` — он уже мёртв
- Убрать: починить родителя (`wait()`) или убить родителя (тогда зомби усыновит systemd и уберёт)

### fork() и wait():
- `fork()` — создать копию текущего процесса (дочерний)
- `wait()` — забрать код возврата дочернего, удалить запись

Если родитель умер раньше дочернего — дочерний становится сиротой, усыновляет `init`/`systemd` (PID 1) который сразу вызывает `wait()`.

## 5.6. Сигнал HUP демону

```bash
kill -SIGHUP $(pidof nginx)
# или через systemd:
systemctl reload nginx
```

**HUP** — hangup. Исторически = "терминал отключился". Демоны переопределили его как "перечитай конфиг без перезапуска".

### Graceful reload vs restart:
```bash
systemctl restart nginx   # стоп + старт = даунтайм
systemctl reload nginx    # SIGHUP = без даунтайма
```

При SIGHUP nginx:
1. Читает новый конфиг
2. Запускает новые worker процессы
3. Старые доделывают текущие запросы и умирают

### Демон (daemon):
Процесс работающий в фоне без терминала. Имя обычно заканчивается на `d` (nginx**d**, ssh**d**, system**d**). stdin/stdout/stderr → `/dev/null`.

## 5.7. Job control в неинтерактивном скрипте

**Интерактивный терминал** — job control включён: `Ctrl+Z`, `fg`, `bg`, `jobs`.

**Скрипт** — job control **выключен по умолчанию**. Используют `&` + `wait`:

```bash
./long_process.sh &   # запустить в фоне
PID=$!                # $! — PID последнего фонового процесса
wait $PID             # подождать завершения
```

Включить принудительно (редко нужно):
```bash
set -m   # monitor mode
```

Проверить интерактивность:
```bash
[[ $- == *i* ]] && echo "интерактивный" || echo "неинтерактивный"
```

## 5.8. Потоки (LWP) процесса

```bash
ps -p 1234 -L         # -p = PID, -L = показать LWP (потоки)
ls /proc/1234/task/   # каждая директория = отдельный поток
```

**LWP** — Light Weight Process. В Linux потоки = процессы с общей памятью.

```
процесс — своя память, своё адресное пространство
поток   — общая память, свой стек
```

```bash
ps -p 1234 -L
# PID   LWP  ...
# 1234  1234  # главный поток
# 1234  1235  # поток 2
# 1234  1236  # поток 3
```

### Процессы vs потоки:
| | Процессы | Потоки |
|--|----------|--------|
| Память | Изолирована | Общая |
| Падение одного | Остальные живут | Все падают |
| Обмен данными | Медленнее (IPC) | Быстро (общая память) |
| Пример | nginx workers | рендеринг |

## 5.9. Утилиты мониторинга

**`top`** — стандартная, есть везде:
```
q — выйти
k — kill процесс
r — renice
M — сортировать по памяти
P — сортировать по CPU
```

**`htop`** — расширенная, надо устанавливать:
- Цветной интерфейс, графики по каждому ядру отдельно
- Мышь работает
- F9 — послать сигнал, F6 — сортировка
- Дерево процессов

Ключевое отличие: `htop` показывает загрузку каждого ядра отдельно — видно если один поток грузит одно ядро на 100% пока остальные простаивают.

## 5.10. Приостановить, фон, передний план

```bash
Ctrl+Z          # SIGTSTP → заморозить процесс
bg              # продолжить последний job в фоне
fg              # вернуть последний job на передний план
fg %2           # вернуть job номер 2
jobs            # список всех jobs
```

Полный цикл:
```bash
ping google.com    # запустили (бесконечно, -c N для N пакетов)
Ctrl+Z             # заморозили
bg                 # продолжили в фоне
fg                 # вернули
Ctrl+C             # убили
```

## 5.11. SIGTERM → подождать → SIGKILL

```bash
kill -SIGTERM 1234
sleep 5
kill -0 1234 2>/dev/null && kill -SIGKILL 1234
```

- `kill -SIGTERM` — вежливая просьба, процесс может почиститься (закрыть файлы, БД, lock-файлы)
- `sleep 5` — даём время
- `kill -0` — проверить жив ли (не посылает сигнал, только проверяет)
- `kill -SIGKILL` — принудительно если не завершился
- `2>/dev/null` — подавить ошибку если процесс уже умер сам
