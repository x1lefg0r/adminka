## 1.1. Как оболочка интерпретирует строку

Bash — это интерпретатор языка. Когда вводишь строку, он не просто "запускает команду" — он парсит её и строит **AST (дерево)**.

### Этапы обработки (слева направо внутри каждого этапа):

1. **Токенизация** — разбивка на слова и операторы
   - Пробел = разделитель токенов
   - Спецсимволы (`|`, `>`, `<`, `&`, `;`, `(`, `)`) — операторы, не часть слов
   - `|` сразу делит строку на командные сегменты

2. **Раскрытие** — переменные, подстановки команд
   - `$VAR` → значение переменной
   - `$(cmd)` → вывод команды (подстановка команды, не то же что просто запуск)
   - `~` → домашняя директория

3. **Word splitting** — повторная разбивка результатов подстановки по пробелам
   - Срабатывает на результатах `$VAR` и `$(cmd)` если они без кавычек
   - `"$VAR"` — отключает word splitting для этого значения

4. **Glob** — сопоставление шаблонов с файловой системой
   - `*` — любое количество любых символов
   - `?` — ровно один символ
   - `**` — рекурсивно (только если включён `globstar`)
   - Оболочка идёт в ФС, ищет совпадения, подставляет список файлов
   - Если совпадений нет — передаёт шаблон буквально (баг-ловушка)

5. **Запуск**

### Пример с pipe:
```bash
ls | grep foo
```
- Токенизация: видит `|`, делит на два сегмента `[ls]` и `[grep, foo]`
- Строит дерево:
  ```
  pipeline
  ├── command: ls
  └── command: grep
          └── arg: foo
  ```
- При запуске: создаёт pipe (буфер в ядре), форкает два процесса, подменяет им файловые дескрипторы

### Экранирование:
```bash
echo \*        # выведет *, не список файлов
echo '$HOME'   # выведет $HOME буквально
echo "\$HOME"  # то же через двойные кавычки
```

## IFS — Internal Field Separator

IFS определяет по каким символам bash режет строку на токены на этапе word splitting.

По умолчанию: пробел + таб + newline:
```bash
str="one two three"
for word in $str; do echo $word; done
# → one
# → two
# → three
```

Меняешь IFS — меняется поведение всей оболочки:
```bash
IFS=:
for part in $PATH; do echo $part; done
# → /usr/local/bin
# → /usr/bin
# → /bin
```

Удобно парсить PATH, /etc/passwd, любые CSV-подобные строки.

Всегда восстанавливай после изменения:
```bash
OLD_IFS=$IFS
IFS=:
# делаем что нужно
IFS=$OLD_IFS

# или через subshell — не трогает текущую оболочку:
(IFS=: && for part in $PATH; do echo $part; done)
```

`IFS=` (пустой) — отключает word splitting полностью. Используется в безопасном чтении файлов:
```bash
while IFS= read -r line; do
    echo "$line"
done < file.txt
```

## 1.2. Переменные окружения vs локальные переменные

```bash
LOCAL=hello          # локальная — только в текущей оболочке, не передаётся дочерним
export ENV=hello     # переменная окружения — передаётся дочерним процессам
```

### Ключевые моменты:
- `export` — не создаёт переменную, а **помечает** её флагом "передавать дочерним"
- Внутри bash хранится как `{ value: "hello", exported: true }`
- Дочерний получает **копию** — изменения не влияют на родителя
- Наследование только **вниз**, никогда вверх
- Соседние скрипты не знают друг о друге — у каждого своя копия от родителя
- `export -n VAR` — снять флаг экспорта

```bash
# parent.sh
export VAR=hello
./child.sh       # child меняет VAR=world внутри себя
echo $VAR        # → hello, родитель не знает об изменении
```

## 1.3. Алгоритм поиска исполняемого файла

Порядок поиска при вводе команды:

1. **Алиасы** — есть ли `alias ls=...`?
2. **Функции** — есть ли функция с таким именем?
3. **Builtins** — встроенные команды bash (`cd`, `echo`, `export` и др.) — не файлы на диске
4. **PATH** — по директориям слева направо, ищет файл с битом исполнения `x`

```bash
which ls        # где нашёл в PATH
type ls         # что именно — alias/function/builtin/file
type cd         # → cd is a shell builtin
```

### PATH — как работает:
```bash
echo $PATH
# → /usr/local/bin:/usr/bin:/bin:/usr/sbin
```
- Строка режется по `:` (захардкожено в bash)
- Для каждой директории проверяет: существует ли файл с таким именем и битом `x`?
- Нашёл — запускает, дальше не идёт
- Поиск **не рекурсивный** — только в самой директории, не в подкаталогах

### Если в имени есть `/`:
```bash
/bin/ls          # абсолютный путь — PATH не смотрит вообще
./myscript.sh    # то же самое
```

### Принудительно обойти алиасы/функции:
```bash
\ls              # экранирование отключает alias
command ls       # искать только в PATH
builtin cd       # только builtin
```

### Команды в PATH — что за файлы:
```bash
file /usr/bin/ls   # → ELF 64-bit executable (скомпилированный бинарник)
```
Либо бинарник (ELF), либо скрипт с shebang (`#!/bin/bash`). Главное — бит `x`.

<a id="q-1-5"></a>
## 1.4. Aliases vs функции

### Alias — простая замена строки:
```bash
alias ll='ls -la'
alias gs='git status'
```
- Раскрывается **до токенизации** — раньше всего остального
- Живёт в памяти сессии, прописывают в `~/.bashrc`
- Аргументы прилепляются автоматически в конец
- **Не передаётся** дочерним процессам даже через `export`

### Функция — полноценный код:
```bash
mkcd() {
    mkdir -p "$1"
    cd "$1"
}
```
- Живёт в памяти сессии
- Можно передать дочерним: `export -f mkcd`
- Имеет свой контекст: `$1`, `$2`, локальные переменные

### Когда функция строго обязательна:
```bash
# 1. Аргументы в произвольном месте (алиас не может)
backup() { cp "$1" "$1.bak"; }

# 2. Несколько команд с логикой
deploy() { cd /app && git pull && npm install; }

# 3. Условия, циклы
retry() {
    for i in 1 2 3; do
        "$@" && return 0
        sleep 1
    done
}

# 4. Локальные переменные
calc() {
    local result=$(( $1 + $2 ))
    echo $result
}
```

### Почему не переменная вместо алиаса:
```bash
VAR='ls -la'
$VAR       # иногда работает через word splitting, но костыль
"$VAR"     # не сработает — ищет команду "ls -la" целиком
```
Переменная хранит данные, алиас — замена команды.

## 1.6. $HOME и текущий каталог в одной строке

```bash
echo "$HOME $(pwd)"
# или без дочернего процесса:
echo "$HOME $PWD"
```

### Способы получить домашнюю директорию:
```bash
echo $HOME          # переменная окружения
echo ~              # glob раскрытие
echo ~username      # домашняя директория конкретного пользователя
echo ~root          # → /root
getent passwd $USER | cut -d: -f6  # напрямую из /etc/passwd
```

`$HOME` берётся из `/etc/passwd` при логине. `$PWD` — переменная которую bash обновляет при каждом `cd`.

## 1.7. History expansion

```bash
!!     # вся предыдущая команда
!^     # первый аргумент === !:1
!$     # последний аргумент === !:-1
!*     # все аргументы === !:1-$
```

### Синтаксис токенов (`!:N`):
```bash
rm -rf /tmp/test
# токен 0: rm
# токен 1: -rf
# токен 2: /tmp/test

!:0    # → rm
!:1    # → -rf
!:2    # → /tmp/test
!:1-2  # → -rf /tmp/test (диапазон)
!:0-$  # === !! (вся команда)
```

`^` = начало, `$` = конец (как в regex).

### Проверить не запуская:
```bash
echo !!   # покажет что раскроется
```

### Опасность:
- В интерактивном терминале можно случайно повторить деструктивную команду
- В скриптах history expansion **выключен по умолчанию**
- Опасно в паре с `eval`

### Другие примеры:
```bash
sudo !!          # повторить предыдущую команду с sudo
cd !$            # перейти в последний аргумент предыдущей команды
!:0-$            # вся команда целиком
```

## 1.8. PATH и временное добавление каталога

```bash
export PATH="/my/dir:$PATH"   # в начало — приоритет выше
export PATH="$PATH:/my/dir"   # в конец — приоритет ниже
```

`$PATH` раскрывается сразу в текущее значение — никакой рекурсии, просто конкатенация строк.

**Временно** = только текущая сессия. Постоянно = в `~/.bashrc`.

### На один запуск:
```bash
PATH="/my/dir:$PATH" ./script.sh   # только для этой команды
```

### Никогда не добавляй `.` в PATH:
```bash
export PATH=".:$PATH"   # опасно — файл ls в текущей директории выполнится вместо системного
```

<a id="q-1-10"></a>
## 1.9. Одинарные vs двойные кавычки

### Двойные кавычки — раскрывается:
- `$VAR` — переменные
- `$(cmd)` — подстановка команд
- `\\` — экранирование

### Двойные кавычки — НЕ раскрывается:
- Глобы `*`, `?`
- Word splitting отключён для содержимого

### Одинарные кавычки — абсолютно всё буквально:
```bash
VAR=world
echo "$VAR"    # → world
echo '$VAR'    # → $VAR
echo "$(date)" # → Mon Jun...
echo '$(date)' # → $(date)
echo '\n'      # → \n буквально
```

**Правило:** всегда бери переменные в двойные кавычки `"$VAR"` — защита от word splitting и glob.

<a id="q-1-11"></a>
## 1.10. Поиск в истории без интерактивного поиска

```bash
history | grep "нужная строка"
grep "нужная строка" ~/.bash_history
```

### `history` vs `~/.bash_history`:
- `history` — история текущей сессии в памяти (включая незаписанные команды)
- `~/.bash_history` — файл на диске, пишется **при закрытии сессии**
- Если терминал упал — текущая сессия в файл не запишется

### Выполнить по номеру:
```bash
history | grep "git"
# → 342  git commit -m "fix"
!342   # выполнит команду номер 342
```

## 1.11. Все переменные окружения по алфавиту

```bash
env | sort        # только переменные окружения
printenv | sort   # то же самое
set | sort        # всё: окружение + локальные + функции (намного больше)
```

### `env` умеет запускать команды с изменённым окружением:
```bash
env VAR=hello ./script.sh        # прокинуть переменную на один запуск
env -i PATH=/bin ./script.sh     # запустить с чистым окружением
```
Текущее окружение не трогается — VAR появляется только внутри скрипта.

## 1.12. Ctrl+C, Ctrl+D, Ctrl+Z

| Комбинация | Сигнал | Что происходит |
|------------|--------|----------------|
| `Ctrl+C` | SIGINT | Прерывает процесс. Процесс **может поймать и обработать** |
| `Ctrl+D` | — | Не сигнал. Закрывает stdin (EOF). Программа получает "данных больше нет" и завершается |
| `Ctrl+Z` | SIGTSTP | Приостанавливает процесс, отправляет в background. Процесс не умирает |

### После Ctrl+Z:
```bash
jobs        # список приостановленных процессов
fg          # вернуть на передний план
bg          # продолжить в фоне
kill %1     # убить первый job
```

### Ctrl+D подробнее:
Программы читают stdin в цикле. EOF = "ввод закончился" → программа завершает цикл. Поэтому `cat`, `python3`, сам bash — все завершаются на Ctrl+D.

### Если процесс игнорирует Ctrl+C:
```bash
trap '' SIGINT   # скрипт игнорирует SIGINT
```
Тогда используй `Ctrl+\` (SIGQUIT) — убивает и создаёт **core dump** (снимок памяти процесса на диске для отладки).

### trap:
```bash
trap 'команда' СИГНАЛ              # выполнить при сигнале
trap '' SIGINT                     # игнорировать
trap - SIGINT                      # сбросить в дефолт
trap 'rm -f /tmp/lock; exit' SIGINT SIGTERM  # почистить перед выходом
```

### eval (опасная штука):
```bash
eval "$cmd"   # выполняет строку как команду, прогоняет через полный парсер bash
```
Нужен когда строка содержит операторы (`|`, `;`). Опасен с пользовательским вводом:
```bash
user_input="foo; rm -rf /"
eval "echo $user_input"   # выполнит rm -rf /
```
**Правило:** никогда не используй `eval` с пользовательским вводом.

## 2.1. Inode

Inode — структура данных в файловой системе которая хранит **метаданные файла**, но не его имя и не содержимое.

### Что хранится в inode:
- Права доступа (`rwxr-xr-x`)
- Владелец (UID, GID)
- Размер файла
- Временные метки (atime, mtime, ctime)
- Указатели на блоки данных на диске
- Счётчик жёстких ссылок

### Что НЕ хранится:
- Имя файла — оно хранится в директории
- Содержимое — оно в блоках данных на диске

```bash
ls -i file.txt    # показать номер inode
stat file.txt     # все метаданные inode
stat -x file.txt  # на macOS
stat -c '%h' file.txt    # только счётчик ссылок
stat -c '%i' file.txt    # только номер inode
stat -c '%n %i %h' file.txt  # имя + inode + ссылки
```

### Блоки данных:
Диск разбит на блоки фиксированного размера — обычно 4096 байт (4KB). Inode хранит список номеров блоков, ядро читает их по порядку и получает данные.

```bash
ls -lh file.txt   # размер файла — реальные байты
du -h file.txt    # занято на диске — кратно размеру блока
```

### Правильная модель файла в Linux:

`touch file.txt` делает две вещи:
1. Создаёт inode (выделяет метаданные + блоки)
2. Создаёт запись в директории `"file.txt" → inode 12345`

**В Linux нет "файлов" в привычном смысле. Есть inode-ы и имена которые на них указывают.**

Директория — это просто таблица `имя → номер inode`.

```
"file.txt" ──→ inode 12345 ──→ [указатели на блоки] ──→ [блок 4521][блок 4522]
                                                          "hello world"
```

## 2.2. Жёсткие vs символические ссылки

```bash
ln file.txt hardlink      # жёсткая ссылка
ln -s file.txt symlink    # символическая ссылка
```

| | Жёсткая | Символическая |
|--|---------|---------------|
| Что это | Ещё одно имя для того же inode | Отдельный файл содержащий путь |
| Удалить оригинал | Данные живут пока счётчик > 0 | Ссылка становится битой |
| Между ФС | Нельзя (inode номера уникальны только внутри ФС) | Можно |
| На директорию | Нельзя (риск циклов) | Можно |

### Счётчик жёстких ссылок:
```bash
echo "hello" > file.txt   # счётчик = 1
ln file.txt hard1         # счётчик = 2
ln file.txt hard2         # счётчик = 3
rm file.txt               # счётчик = 2, данные живут
rm hard1                  # счётчик = 1, данные живут
rm hard2                  # счётчик = 0, данные удаляются с диска
```

**Правильное название удаления в Linux — `unlink`.** Ты не удаляешь данные — отвязываешь имя от inode. Данные исчезают когда счётчик = 0.

### Жёсткая ссылка — нет оригинала и копии:
```bash
ln file.txt hard
rm file.txt   # "оригинал" удалён
cat hard      # данные живы — hard такой же полноправный владелец
```

```
"file.txt" ─┐
             ├──→ inode 12345 ──→ данные на диске
"hard"     ─┘
```

Два указателя на одну структуру. Ни один не главнее.

## 2.3. Поиск пустых файлов и удаление

```bash
# сначала посмотреть
find /var/log -type f -empty

# потом удалить
find /var/log -type f -empty -delete
```

### Флаги `-type`:
```bash
-type f   # файлы
-type d   # директории
-type l   # символические ссылки
-type p   # именованные каналы (FIFO)
-type s   # сокеты
```

### Вместо `-empty` — фильтры по размеру:
```bash
-size 0           # пустые
-size +100M       # больше 100MB
-size -1k         # меньше 1KB
-size +1M -size -100M  # между 1MB и 100MB
```

По времени:
```bash
-mtime -1     # изменён меньше суток назад
-mtime +30    # изменён больше 30 дней назад
-newer file   # новее чем file
```

По правам:
```bash
-perm 644
-perm /111    # есть хоть один бит исполнения
```

### `-delete` vs pipe:
```bash
# безопасно с пробелами в именах:
find /var/log -type f -empty -print0 | xargs -0 rm
```

`-print0` разделяет нулевым байтом `\0` вместо `\n`. `-0` в xargs читает по `\0`. Нулевой байт не может быть в имени файла — безопасный разделитель.

### xargs:
Берёт stdin и передаёт как аргументы команде:
```bash
find /var/log -type f -empty | xargs rm
# превращает список файлов в: rm /var/log/a.log /var/log/b.log
```

`rm` не умеет читать stdin — только аргументы. xargs — мост между stdout одной команды и аргументами другой.

<a id="q-2-4"></a>
## 2.4. Временные метки файла

| Метка | Название | Когда обновляется |
|-------|----------|-------------------|
| `atime` | access time | при **чтении** файла |
| `mtime` | modify time | при изменении **содержимого** |
| `ctime` | change time | при изменении **метаданных** (права, владелец) или содержимого |

```bash
stat file.txt
# Access: 2026-06-24 ...   ← atime
# Modify: 2026-06-24 ...   ← mtime
# Change: 2026-06-24 ...   ← ctime
```

### Важные нюансы:
- `ctime` ≠ creation time (частое заблуждение). Время создания в Linux классически не хранится
- `mtime` всегда обновляет `ctime` — изменение содержимого меняет размер в inode (метаданные)
- Они расходятся когда меняешь метаданные без содержимого:

```bash
chmod 755 file.txt   # ctime обновился, mtime НЕ обновился
```

- `make`, `rsync`, бэкапы смотрят на `mtime` — "когда файл последний раз редактировался"

### `atime` часто отключают:
```bash
# в /etc/fstab:
/dev/sda1 / ext4 noatime 0 1
```
Каждое чтение файла вызывает запись на диск (обновление atime) — лишняя нагрузка.

## 2.5. Топ-10 самых больших файлов

```bash
# просто и надёжно:
du -ah /path | sort -rh | head -10
```

- `du` — disk usage
- `-a` — all, включить файлы (по умолчанию только директории)
- `-h` — human readable (KB/MB/GB)
- `sort -rh` — `-r` reverse, `-h` понимает суффиксы KB/MB/GB
- `head -10` — первые 10

```bash
# точнее — сортировка по байтам:
find /path -type f -printf '%s %p\n' | sort -rn | head -10 | awk '{print $2}' | xargs du -sh
```

- `-printf '%s %p\n'` — встроенный форматированный вывод find (не запускает внешний процесс)
  - `%s` — размер в байтах
  - `%p` — путь
- `sort -rn` — `-r` reverse, `-n` numeric (сортировать как числа, не строки)
- `awk '{print $2}'` — взять второй столбец (путь), отбросить размер в байтах

### awk:
**AWK** — Aho, Weinberger, Kernighan (фамилии авторов). Язык обработки текста.

```bash
awk '{print $2}'   # для каждой строки напечатать второе поле
#   ↑            ↑
#   одинарные кавычки bash — чтобы $2 не раскрылся как переменная bash
```

Внутри awk: `{}` — блок на каждую строку, `$1/$2/...` — поля разделённые пробелом.

<a id="q-2-6"></a>
## 2.6. Почему удаление файла не освобождает место

У inode два счётчика:
- **hardlink count** — количество имён в директориях
- **open file descriptor count** — количество процессов держащих файл открытым

Место освобождается только когда **оба счётчика = 0**.

```bash
# процесс пишет в nginx.log
rm nginx.log        # hardlink count = 0, но fd count = 1 (nginx держит открытым)
                    # место НЕ освободилось, nginx продолжает писать
df -h               # место всё ещё занято
```

### Найти такие файлы:
```bash
lsof | grep deleted   # открытые но удалённые файлы
```

### Освободить место без перезапуска процесса:
```bash
> /proc/PID/fd/3   # обнулить через файловый дескриптор процесса
```

### lsof — list open files:
Показывает все открытые файлы в системе прямо сейчас (файлы, сокеты, пайпы — в Linux всё файл).

```bash
lsof
# COMMAND  PID  USER  FD   TYPE  DEVICE  SIZE  NODE  NAME
# nginx   1234  root  3u   REG   sda1    512   4521  /var/log/nginx.log (deleted)
```

Столбцы:
- `FD` — номер дескриптора + режим (`r`=read, `w`=write, `u`=read+write)
- `TYPE` — `REG`=файл, `DIR`=директория, `IPv4`=сокет, `FIFO`=пайп, `CHR`=символьное устройство
- `NODE` — номер inode

```bash
lsof -p 1234     # -p = PID, все файлы процесса
lsof /var/log    # кто держит файлы в директории
lsof -i :80      # -i = internet, кто держит порт 80
lsof -u god      # -u = user, все файлы пользователя
lsof -i TCP      # все TCP соединения
lsof -i @1.2.3.4 # соединения с конкретным IP
```

### Файловые дескрипторы процесса:
```
0 → stdin
1 → stdout
2 → stderr
3, 4, 5... → файлы открытые в ходе работы (следующий свободный номер)
```

Каждый процесс имеет свою таблицу дескрипторов. Дескриптор `3` у процесса 1234 и `3` у процесса 5678 — разные, указывают на разные файлы.

### /proc/PID/fd/:
```bash
ls -la /proc/1234/fd/
# 0 → /dev/null
# 3 → /var/log/nginx.log (deleted)
```

Через дескриптор в `/proc` можно добраться до данных удалённого файла пока процесс его держит открытым (вопрос 2.12).

### Процесс:
Запущенная программа в памяти:
```
процесс = код (инструкции)
        + данные (переменные, heap)
        + стек (вызовы функций)
        + таблица дескрипторов (открытые файлы)
        + метаданные (PID, владелец, приоритет...)
```

Один бинарник можно запустить несколько раз — несколько независимых процессов с разными PID.

## 2.9. Копирование vs перемещение в пределах одной ФС

**`mv` в пределах одной ФС:**
- Создаёт новую запись в директории назначения
- Удаляет старую запись
- Inode не трогается, данные не копируются
- **Мгновенная операция** независимо от размера

**`cp`:**
- Создаёт новый inode
- Выделяет новые блоки
- Копирует данные побайтово
- Время зависит от размера

**`mv` между разными ФС:**
Вынужден делать `cp` + `rm` — уже не мгновенно.

### Что такое разные ФС:
```bash
df -h
# /dev/sda1   50G  /          # корневая ФС
# /dev/sda2   20G  /home      # отдельная ФС
# /dev/sdb1  500G  /data      # отдельный диск
# /dev/sdc1  100G  /var/log   # даже вложенная папка может быть отдельной ФС
```

Файловая система — раздел диска примонтированный в точку. Не обязательно на корневом уровне.

```bash
df /home/god/file.txt /data/   # проверить на каких ФС лежат пути
```

## 2.10. Что происходит с символической ссылкой при удалении цели

Ссылка становится **битой** — файл с путём который никуда не ведёт. Сама ссылка не удаляется автоматически.

```bash
echo "hello" > file.txt
ln -s file.txt symlink
rm file.txt

cat symlink    # → No such file or directory
ls -la symlink # → symlink -> file.txt  (ссылка есть, цели нет)
```

Жёсткая ссылка при удалении "оригинала" — данные живы:
```bash
ln file.txt hardlink
rm file.txt
cat hardlink   # → hello
```

## Шпаргалка по перенаправлениям

```
>    # stdout → файл (перезапись)
>>   # stdout → файл (добавить в конец)
<    # файл → stdin
<<<  # строка → stdin (here-string)
<<   # here-document → stdin (многострочно)
2>   # stderr → файл
2>>  # stderr → файл (добавить)
2>&1 # stderr → туда же куда stdout
&>   # stdout + stderr → файл (перезапись)
&>>  # stdout + stderr → файл (добавить)
```

Логика стрелок: куда смотрит стрелка — туда идёт поток.

<a id="q-3-2"></a>
## 3.1. Файловые дескрипторы 1 и 2

- `1` — stdout, стандартный вывод
- `2` — stderr, стандартный поток ошибок

Почему важно разделять в скриптах:
```bash
ls /существующий /несуществующий
# stdout: /существующий/file.txt
# stderr: ls: /несуществующий: No such file or directory
```

Если не разделять — ошибки мешаются с данными:
```bash
ls /path | grep "something"          # grep получит и вывод и ошибки вперемешку
ls /path 2>/dev/null | grep "something"  # правильно — ошибки выбросить
```

### `/dev/null` — чёрная дыра:
Всё что туда пишешь — исчезает. Читать из него — сразу EOF.

Антипаттерн — заглушать всё подряд:
```bash
command 2>/dev/null   # если команда упала — никогда не узнаешь почему
```

<a id="q-3-3"></a>
## 3.2. Добавить stdout и stderr в один лог-файл

```bash
command >> log.txt 2>&1   # правильно
command 2>&1 >> log.txt   # НЕПРАВИЛЬНО — stderr остаётся на терминале
```

Порядок важен: сначала stdout в файл, потом stderr туда же куда смотрит stdout.

```bash
command &>> log.txt   # синтаксический сахар, то же самое
```

### `2>&1` — идея:
`>&` означает "перенаправить дескриптор в другой дескриптор":
```bash
2>&1   # stderr → туда же куда stdout
1>&2   # stdout → туда же куда stderr
```

`&` — disambiguator, говорит bash что `1` это номер дескриптора, не имя файла:
```bash
2>1    # перенаправить stderr в файл с именем "1"
2>&1   # перенаправить stderr в дескриптор 1
```

## 3.3. Как конвейер (pipe) обрабатывает поток данных

```bash
ls | grep foo
```

Ядро создаёт pipe — буфер в памяти (обычно 64KB) с двумя концами:
- пишущий конец → stdout `ls`
- читающий конец → stdin `grep`

Оба процесса запускаются **одновременно**, не последовательно.

### Если принимающий процесс завершается раньше:
```bash
ls -la | head -5   # head читает 5 строк и выходит
```
`ls` продолжает писать → pipe заполнен → `ls` получает **SIGPIPE** → завершается. Нормальное поведение, не ошибка. Именно так `head`, `less`, `grep` останавливают бесконечные потоки.

<a id="q-3-5"></a>
## 3.4. Here-string и Here-document

### Here-string (`<<<`):
```bash
grep "foo" <<< "$VAR"   # передать переменную как stdin
```

Без лишнего процесса в отличие от `echo "$VAR" | grep "foo"`.

```bash
# read из строки:
read a b c <<< "one two three"
echo $a   # → one

# калькулятор:
bc <<< "2 + 2"   # → 4
```

### Here-document (`<<`):
```bash
cat << EOF
строка 1
строка 2
EOF
```

`EOF` — просто метка, можно любое слово. Закрывающая метка — одна на строке без пробелов.

```bash
cat << КОНЕЦ
строка
КОНЕЦ

# с табами (<<-) — игнорирует ведущие табы:
cat <<- EOF
	строка   # таб игнорируется
	EOF      # таб перед меткой тоже игнорируется
```

Реальный сценарий:
```bash
ssh user@host << EOF
cd /var/log
tail -100 nginx.log
EOF
```

## 3.5. Pipe vs Process substitution

Pipe — один поток. `diff` нужно два входа — pipe не справится.

```bash
diff <(ls /dir1) <(ls /dir2)   # process substitution
```

`<(cmd)` — запускает команду, вывод делает доступным как временный файл через `/dev/fd/N`. Команда думает что получила файл, на самом деле поток.

```bash
# pipe — один поток, последовательно:
cmd1 | cmd2

# process substitution — несколько потоков одновременно:
diff <(cmd1) <(cmd2)
```

<a id="q-3-8"></a>
## 3.7. Именованный канал (FIFO)

Обычный pipe — анонимный, только между родственными процессами (форк).

FIFO — pipe как файл в файловой системе, между любыми несвязанными процессами:

```bash
mkfifo /tmp/mypipe        # создать
ls -la /tmp/mypipe        # → prw-r--r-- (p = pipe)
```

```bash
# терминал 1:
cat /tmp/mypipe           # ждёт данных, блокируется

# терминал 2:
echo "hello" > /tmp/mypipe   # → терминал 1 вывел "hello"
```

Сценарий: два независимых процесса обмениваются данными через FIFO без знания друг о друге.

## 3.9. Фильтр с выводом на экран и в файл

```bash
ls | grep "conf" | tee filtered.txt
```

### grep:
```bash
grep "conf"        # case-sensitive поиск подстроки
grep -i "conf"     # ignore case
grep -v "conf"     # invert — строки которые НЕ содержат
grep -n "conf"     # number — показать номера строк
grep -r "conf" /etc   # recursive — по директории
grep -E "foo|bar"     # extended regex
```

## 4.1. `$@` vs `$*` в двойных кавычках

```bash
args_at "foo bar" "baz"
# "$@" → каждый аргумент отдельно:
# arg: foo bar
# arg: baz

args_star "foo bar" "baz"
# "$*" → все склеены в одну строку:
# arg: foo bar baz
```

- `"$@"` — каждый аргумент отдельная строка, пробелы сохраняются
- `"$*"` — все аргументы склеены через первый символ `$IFS` (обычно пробел)
- Без кавычек — одинаковы, оба делают word splitting

Функция вызывается **один раз** — цикл `for` внутри итерируется по элементам.

**Правило:** всегда используй `"$@"` когда передаёшь аргументы дальше.

## 4.2. Значение по умолчанию через расширение параметров

```bash
${VAR:-default}   # если VAR не задана ИЛИ пустая → использовать "default"
${VAR-default}    # если VAR не задана (но не если пустая) → использовать "default"
${VAR:=default}   # если VAR не задана ИЛИ пустая → присвоить "default" и использовать
${VAR:?error}     # если VAR не задана ИЛИ пустая → вывести ошибку и выйти
${VAR:+other}     # если VAR задана и не пустая → использовать "other" вместо неё
```

Логика:
```
${  VAR  :-  default  }
    ↑    ↑   ↑
    имя  |   значение если условие выполнено
         модификатор (- = читать, = = присвоить, ? = ошибка, + = инверсия)
```

`:` добавляет проверку на пустоту поверх проверки на существование:
```bash
VAR=""
echo ${VAR:-default}   # → default (пустая → сработало)
echo ${VAR-default}    # → ""      (задана, пусть пустая → не сработало)
```

На практике:
```bash
NAME=${1:-"anonymous"}   # первый аргумент или дефолт
DIR=${DIR:-"/tmp"}       # переменная или дефолт

# добавить суффикс только если переменная задана:
DEBUG=1
echo "running${DEBUG:+ in debug mode}"   # → running in debug mode
```

## 4.3. Безопасный перебор файлов с пробелами

```bash
find /path -type f -print0 | while IFS= read -r -d '' file; do
    echo "обрабатываю: $file"
done
```

- `IFS=` — очищаем разделитель, `read` не режет по пробелам
- `read -r` — не интерпретировать `\` как экранирование
- `-d ''` — читать до нулевого байта (`\0`)
- `-print0` — find разделяет нулевым байтом

### IFS — Internal Field Separator:
```bash
IFS=: read a b c <<< "one:two:three"
echo $a   # → one

IFS= read a <<< "one two three"
echo $a   # → one two three (всё в одну переменную, не режет)
```

Простой вариант если пробелов нет:
```bash
for file in /path/*.txt; do
    echo "$file"
done
```

## 4.4. Ассоциативный массив

```bash
declare -A colors
colors["apple"]="red"
colors["banana"]="yellow"

echo ${colors["apple"]}   # → red
```

Это словарь/хэшмап как в JS: `{ apple: "red", banana: "yellow" }`.

Инициализация сразу:
```bash
declare -A colors=([apple]="red" [banana]="yellow")
```

Операции:
```bash
echo ${!colors[@]}   # все ключи (! = инвертировать — дать ключи вместо значений)
echo ${colors[@]}    # все значения
echo ${#colors[@]}   # количество элементов (# = количество)

for key in "${!colors[@]}"; do
    echo "$key → ${colors[$key]}"
done

unset colors["apple"]   # удалить элемент
```

### `declare` флаги:
```bash
declare -A   # Associative array
declare -a   # обычный индексный array
declare -i   # integer
declare -r   # readonly
declare -x   # export
declare -f   # показать функции
```

## 4.6. Целочисленная арифметика

```bash
result=$(( 2 + 2 ))
echo $result   # → 4
```

Операторы:
```bash
$(( a + b ))   # сложение
$(( a - b ))   # вычитание
$(( a * b ))   # умножение
$(( a / b ))   # целочисленное деление
$(( a % b ))   # остаток
$(( a ** b ))  # степень
```

Внутри `(( ))` переменные без `$`:
```bash
echo $(( a + b ))    # правильно
echo $(( $a + $b ))  # работает но избыточно
```

Инкремент и присвоение:
```bash
(( a++ ))
(( a += 5 ))
(( result = a + b ))   # присвоение без $()
echo $result           # читать — с $
```

Логика: пишешь в переменную — без `$`, читаешь — с `$`. Везде в bash одинаково.

## 4.7. Опции оболочки `-e` и `-u`

```bash
#!/bin/bash
set -euo pipefail   # стандартное начало любого серьёзного скрипта
```

- `set -e` (`-o errexit`) — выйти если любая команда вернула ненулевой код
- `set -u` (`-o nounset`) — необъявленная переменная = ошибка (без этого просто пустая строка — молчаливый баг)
- `set -o pipefail` — код возврата конвейера = код последней упавшей команды

```bash
# без pipefail:
cat /несуществующий | grep "foo"   # grep вернул 0 → set -e не сработает
# с pipefail:
cat /несуществующий | grep "foo"   # → ошибка, скрипт остановится
```

Включить/выключить:
```bash
set -e    # включить
set +e    # выключить
```

Посмотреть все опции:
```bash
set -o    # все опции и их состояние
```

## 4.8. Проверка строки на регулярное выражение

```bash
if [[ $string =~ ^[a-z]+[0-9]+$ ]]; then
```

```
*      # ноль или более
?      # ноль или одно
^      # начало строки
$      # конец строки
```

## 4.9. `[ ]` vs `[[ ]]`

```bash
[ ]    # старый POSIX, внешняя команда /usr/bin/[
[[ ]]  # bash builtin, расширенный
```

Преимущества `[[ ]]`:

```bash
# 1. Не нужны кавычки (word splitting отключён):
VAR="hello world"
[ $VAR = "hello" ]    # ошибка — word splitting даёт два токена
[[ $VAR = "hello" ]]  # работает

# 2. Логические операторы внутри:
[ $a = "foo" ] && [ $b = "bar" ]
[[ $a = "foo" && $b = "bar" ]]

# 3. Regex:
[[ $str =~ ^[0-9]+$ ]]

# 4. Glob паттерны:
[[ $str == *.txt ]]

# 5. Безопасен с пустыми переменными:
[ -f $FILE ]    # если FILE пустая → синтаксическая ошибка
[[ -f $FILE ]]  # безопасно
```

**Правило:** всегда `[[ ]]` в bash. `[ ]` только для переносимости на sh/dash.

## 4.10. Режим трассировки

```bash
set -x   # включить

```

Показывает каждую команду **после раскрытия переменных** с префиксом `+`:

```bash
# + echo god
```


Аналог автоматического `console.log` на каждой строке. Незаменимо когда скрипт работает локально но падает на сервере.

## Сигналы — шпаргалка

```
Ctrl+C  → SIGINT  (2)  — мягкое прерывание, можно поймать
Ctrl+\  → SIGQUIT (3)  — жёстче + core dump, можно поймать
Ctrl+Z  → SIGTSTP (20) — приостановить, можно поймать

SIGTERM (15) — мягкое завершение, можно поймать (kill по умолчанию)
SIGHUP  (1)  — перечитать конфиг (для демонов)
SIGKILL (9)  — принудительное, НЕЛЬЗЯ поймать или игнорировать
```

```bash
kill 1234          # === kill -SIGTERM 1234
kill -9 1234       # === kill -SIGKILL 1234
kill -0 1234       # проверить существует ли процесс (не посылает сигнал)
```

## 5.1. Uninterruptible sleep (состояние D)

```bash
ps aux
# STAT = D   ← uninterruptible sleep
```

Процесс ждёт завершения операции ввода-вывода и **не может быть прерван сигналами** — даже SIGKILL не поможет. Ядро держит его пока операция не завершится.

Типичные причины:
- Ожидание диска (медленный NFS, зависший диск)
- Ожидание ядерного ресурса
- Баг в драйвере

Единственный способ избавиться — дождаться завершения операции или перезагрузить систему.

<a id="q-5-4"></a>
## 5.3. nice vs приоритеты реального времени

Ядро не может запускать все процессы одновременно — планировщик решает кому дать CPU прямо сейчас. Приоритеты говорят планировщику кто важнее.

### nice — обычный приоритет:
```bash
# значения от -20 (высший) до +19 (низший)
nice -n 19 tar -czf backup.tar.gz /data   # низкий приоритет — не мешает серверу
nice -n -20 ./critical                     # высокий приоритет — только root
renice -n 5 -p 1234                        # изменить приоритет живого процесса
```

Логика названия: чем "nice" (вежливее) процесс — тем больше уступает, тем выше число.

### RT приоритеты — выше всех nice:
```bash
chrt -f 50 ./audio_daemon   # SCHED_FIFO, приоритет 1-99
chrt -r 50 ./critical       # SCHED_RR (round-robin)
```

Используют для критичных к задержкам задач (аудио, промышленные системы).

<a id="q-5-6"></a>
## 5.5. Процесс-зомби

Когда процесс завершается — ядро не удаляет его сразу. Оставляет запись со статусом `Z` пока родитель не вызовет `wait()` и не заберёт код возврата.

```bash
ps aux | grep Z   # найти зомби
```

Аналогия: сотрудник вернулся с отчётом и стоит в дверях. Пока не примешь отчёт — не уйдёт. Стоит в дверях = зомби.

- Зомби не потребляют CPU/память — только запись в таблице
- Нельзя убить через `kill -9` — он уже мёртв
- Убрать: починить родителя (`wait()`) или убить родителя (тогда зомби усыновит systemd и уберёт)

### fork() и wait():
- `fork()` — создать копию текущего процесса (дочерний)
- `wait()` — забрать код возврата дочернего, удалить запись

Если родитель умер раньше дочернего — дочерний становится сиротой, усыновляет `init`/`systemd` (PID 1) который сразу вызывает `wait()`.

## 5.6. Сигнал HUP демону

```bash
kill -SIGHUP $(pidof nginx)
# или через systemd:
systemctl reload nginx
```

**HUP** — hangup. Исторически = "терминал отключился". Демоны переопределили его как "перечитай конфиг без перезапуска".

### Graceful reload vs restart:
```bash
systemctl restart nginx   # стоп + старт = даунтайм
systemctl reload nginx    # SIGHUP = без даунтайма
```

При SIGHUP nginx:
1. Читает новый конфиг
2. Запускает новые worker процессы
3. Старые доделывают текущие запросы и умирают

### Демон (daemon):
Процесс работающий в фоне без терминала. Имя обычно заканчивается на `d` (nginx**d**, ssh**d**, system**d**). stdin/stdout/stderr → `/dev/null`.

## 5.7. Job control в неинтерактивном скрипте

**Интерактивный терминал** — job control включён: `Ctrl+Z`, `fg`, `bg`, `jobs`.

**Скрипт** — job control **выключен по умолчанию**. Используют `&` + `wait`:

```bash
./long_process.sh &   # запустить в фоне
PID=$!                # $! — PID последнего фонового процесса
wait $PID             # подождать завершения
```

Включить принудительно (редко нужно):
```bash
set -m   # monitor mode
```

Проверить интерактивность:
```bash
[[ $- == *i* ]] && echo "интерактивный" || echo "неинтерактивный"
```

## 5.8. Потоки (LWP) процесса

```bash
ps -p 1234 -L         # -p = PID, -L = показать LWP (потоки)
ls /proc/1234/task/   # каждая директория = отдельный поток
```

**LWP** — Light Weight Process. В Linux потоки = процессы с общей памятью.

```
процесс — своя память, своё адресное пространство
поток   — общая память, свой стек
```

```bash
ps -p 1234 -L
# PID   LWP  ...
# 1234  1234  # главный поток
# 1234  1235  # поток 2
# 1234  1236  # поток 3
```

### Процессы vs потоки:
| | Процессы | Потоки |
|--|----------|--------|
| Память | Изолирована | Общая |
| Падение одного | Остальные живут | Все падают |
| Обмен данными | Медленнее (IPC) | Быстро (общая память) |
| Пример | nginx workers | рендеринг |

## 5.11. SIGTERM → подождать → SIGKILL

```bash
kill -SIGTERM 1234
sleep 5
kill -0 1234 2>/dev/null && kill -SIGKILL 1234
```

- `kill -SIGTERM` — вежливая просьба, процесс может почиститься (закрыть файлы, БД, lock-файлы)
- `sleep 5` — даём время
- `kill -0` — проверить жив ли (не посылает сигнал, только проверяет)
- `kill -SIGKILL` — принудительно если не завершился
- `2>/dev/null` — подавить ошибку если процесс уже умер сам

## 6.1. Пространство ядра vs пространство пользователя

```
kernel space — ядро, драйверы, полный доступ к железу
user space   — все пользовательские процессы, ограниченный доступ
```

Почему важно: если бы любой процесс мог напрямую обращаться к железу — один кривой процесс мог бы положить всю систему. Разделение гарантирует изоляцию.

Переход между ними — только через **системные вызовы** (syscall). Это контролируемые "двери" в ядро.

### Как выглядит переход:
```c
write(1, "hello", 5);
// 1. положить аргументы в регистры CPU
// 2. выполнить инструкцию syscall
// 3. CPU → kernel mode
// 4. ядро выполняет операцию
// 5. CPU → user mode
// 6. вернуть результат
```

```bash
strace ls   # посмотреть все syscall-ы которые делает ls
```

<a id="q-6-2"></a>
## 6.2. Последовательность загрузки

```
BIOS/UEFI → GRUB → ядро → initramfs → root ФС → systemd (PID 1)
```

**1. BIOS/UEFI** — POST (Power On Self Test): проверяет RAM, CPU, диски. Ищет загрузочное устройство.

**2. GRUB** (Grand Unified Bootloader) — загрузчик в начале диска. Показывает меню ОС, загружает ядро в память, передаёт параметры и управление ядру.

**3. Ядро** — инициализирует железо, драйверы, монтирует initramfs как временную корневую ФС.

**4. initramfs** — минимальная временная ФС в памяти. Содержит драйверы для доступа к настоящей корневой ФС (RAID, LVM, шифрование).

**5. Настоящая root ФС** — монтируется, initramfs размонтируется.

**6. systemd (PID 1)** — первый пользовательский процесс, запускает все сервисы.

### systemd — PID 1:
```
systemd (PID 1)
├── sshd
├── nginx
│   ├── worker
│   └── worker
└── bash
    └── твоя команда
```

Убить PID 1 нельзя — ядро игнорирует SIGKILL для PID 1. Выключение через:
```bash
systemctl poweroff   # корректное завершение всех сервисов
```

## 6.3. Параметры ядра при загрузке

```bash
cat /proc/cmdline
# → BOOT_IMAGE=/vmlinuz root=/dev/sda1 quiet splash
```

Параметры передаёт GRUB из своего конфига:
```bash
cat /etc/default/grub
# GRUB_CMDLINE_LINUX="quiet splash"
```

Примеры параметров:
```
root=/dev/sda1   # где корневая ФС
quiet            # не выводить сообщения ядра при загрузке
splash           # графическая заставка
init=/bin/bash   # запустить bash вместо systemd (восстановление)
nomodeset        # отключить драйверы видеокарты
ro               # смонтировать корневую ФС только для чтения
```

### Временно изменить через GRUB:
1. При загрузке зажать `Shift` (BIOS) или `Esc` (UEFI)
2. Выбрать пункт, нажать `e`
3. Найти строку с `linux`, изменить параметры
4. `Ctrl+X` — загрузиться (только для этой загрузки)

### Постоянно:
```bash
nano /etc/default/grub
# GRUB_CMDLINE_LINUX="quiet splash твой_параметр"
# GRUB_TIMEOUT=5     # показывать меню 5 секунд
# GRUB_TIMEOUT=0     # не показывать
# GRUB_TIMEOUT=-1    # ждать вечно

update-grub          # Debian/Ubuntu — перегенерировать grub.cfg
grub2-mkconfig -o /boot/grub2/grub.cfg   # RHEL/CentOS
```

<a id="q-6-4"></a>
## 6.4. Загружаемые модули ядра

Ядро не содержит все драйверы сразу — загружает по необходимости как модули (`.ko` файлы).

```bash
lsmod              # список загруженных модулей
modinfo bluetooth  # информация о модуле
```

### Ручная загрузка (`insmod`) — без разрешения зависимостей:
```bash
insmod /path/to/module.ko   # загрузить напрямую
rmmod module_name           # выгрузить
```

### Через штатные утилиты (`modprobe`) — умный, знает зависимости:
```bash
modprobe bluetooth    # загрузить + все зависимости автоматически
modprobe -r bluetooth # выгрузить + ненужные зависимости
```

### Зависимости модулей:
```
bluetooth
└── requires: rfkill
    └── requires: cfg80211
```

`insmod bluetooth` — ошибка если rfkill не загружен.
`modprobe bluetooth` — загрузит cfg80211 → rfkill → bluetooth автоматически.

Граф зависимостей:
```bash
/lib/modules/$(uname -r)/modules.dep
```

`uname -r` — версия текущего ядра. Модули компилируются под конкретную версию ядра.

### Автозагрузка:
```bash
echo "bluetooth" >> /etc/modules
```

## 6.5. strace — трассировка системных вызовов

```bash
strace -o strace.log /bin/ls
```

- `-o` (output) — куда писать вывод
- По умолчанию пишет в stderr (чтобы не мешать stdout программы)

```bash
strace ls                          # все syscall-ы в stderr
strace -p 1234                     # attach к живому процессу (-p = pid)
strace -e trace=open,read,write ls # фильтр по syscall-ам (-e = expression)
strace -e open,read,write ls       # то же самое, trace= дефолтный тип
strace -c ls                       # статистика вызовов (-c = count)
```

### `-e` мини-язык:
```bash
-e trace=open,read,write    # фильтр по syscall-ам
-e signal=SIGTERM           # фильтр по сигналам
-e read=3                   # показывать содержимое fd 3
-e trace=read,signal=SIGTERM # несколько типов вместе
```

```bash
ltrace ls   # вызовы функций библиотек (в отличие от strace — syscall-ов)
```

<a id="q-6-6"></a>
## 6.6. Виртуальная файловая система /proc

`/proc` — не реальная ФС на диске. Ядро генерирует содержимое на лету при чтении, при закрытии — данные нигде не сохраняются.

```bash
cat /proc/cpuinfo      # информация о CPU
cat /proc/meminfo      # использование памяти
cat /proc/uptime       # время работы системы
cat /proc/cmdline      # параметры загрузки
cat /proc/1234/maps    # карта памяти процесса
ls /proc/1234/fd/      # открытые файловые дескрипторы
```

Под капотом при `cat /proc/meminfo`:
1. `cat` делает `open("/proc/meminfo")`
2. Ядро перехватывает
3. Идёт в свои внутренние C структуры (`struct sysinfo`)
4. Форматирует в текст, отдаёт
5. Файла на диске нет

<a id="q-6-7"></a>
## 6.7. Системный вызов vs функция библиотеки C

```
пользовательский код → функция libc → системный вызов → ядро
```

**Функция libc** — обёртка в user space (`/lib/libc.so`):
```c
printf("hello")    // libc
fopen("file.txt")  // libc
malloc(1024)       // libc
```

**Системный вызов** — переход в kernel space:
```c
write(1, "hello", 5)   // syscall
open("file.txt", ...)  // syscall
brk(...)               // syscall
```

Связь:
```c
printf()  →  write()      // форматирование + буферизация → syscall
fopen()   →  open()       // обёртка над syscall
malloc()  →  brk()/mmap() // просит память у ядра
```

В конце цепочки всегда syscall — только ядро умеет работать с железом. Можно писать без libc напрямую через syscall-ы (Go, Rust, embedded).

```bash
strace ls   # только syscall-ы
ltrace ls   # только функции библиотек
```

## 6.8. Динамическое изменение параметров ядра

**sysctl** — system control:

```bash
sysctl -a                          # все параметры (-a = all)
sysctl net.ipv4.ip_forward         # прочитать
sysctl -w net.ipv4.ip_forward=1    # записать (-w = write)
sysctl -n net.ipv4.ip_forward      # только значение (-n = no name)
sysctl -p                          # применить из /etc/sysctl.conf (-p = apply)

# то же через /proc напрямую:
echo 1 > /proc/sys/net/ipv4/ip_forward
```

Структура параметра = путь в `/proc/sys/`:
```
net.ipv4.ip_forward → /proc/sys/net/ipv4/ip_forward
```

### Постоянно:
```bash
echo "net.ipv4.ip_forward=1" >> /etc/sysctl.conf
sysctl -p
```

### Основные параметры:

**Сеть:**
```bash
net.ipv4.ip_forward=1              # маршрутизация пакетов (нужно для Docker, VPN)
net.ipv4.tcp_syncookies=1          # защита от SYN flood
net.core.somaxconn=65535           # очередь входящих соединений
```

**Память:**
```bash
vm.swappiness=10                   # агрессивность использования swap (0-100)
vm.dirty_ratio=15                  # % RAM для грязных страниц
```

**Ядро:**
```bash
kernel.pid_max=65536               # максимальный PID
kernel.dmesg_restrict=1            # запретить обычным пользователям читать dmesg
```

**Файловая система:**
```bash
fs.file-max=2097152                # максимум открытых файлов в системе
fs.inotify.max_user_watches=524288 # максимум отслеживаемых файлов (для IDE)
```

## 6.10. /proc vs /sys

```
/proc — что происходит (процессы, память, статистика ядра)
/sys  — на чём происходит (железо, устройства, драйверы)
```

**/proc:**
```bash
/proc/PID/       # всё о процессе
/proc/meminfo    # память
/proc/cpuinfo    # CPU
/proc/sys/       # параметры ядра (sysctl)
```

**/sys (sysfs)** — структурированный, появился позже как замена части /proc:
```bash
/sys/class/net/eth0/     # сетевой интерфейс
/sys/block/sda/          # блочное устройство
/sys/bus/pci/devices/    # PCI устройства

# пример — изменить яркость:
echo 500 > /sys/class/backlight/intel_backlight/brightness
```

Оба дают прямой доступ к ядру — просто к разным его частям. `/proc` исторически появился раньше, немного хаотичный. `/sys` структурированный, специально для устройств.

## 7.1. Назначения директорий по FHS

**FHS** — Filesystem Hierarchy Standard, стандарт что где должно лежать.

```
/etc     — et cetera — конфигурационные файлы (nginx.conf, sshd_config)
```

```
/bin     — binaries — бинарники
/lib     — libraries — библиотеки
/home    — домашние директории пользователей
/root    — домашняя директория root
/boot    — файлы загрузки (ядро, initramfs, grub)
/dev     — devices — файлы устройств
/proc    — виртуальная ФС процессов
/sys     — виртуальная ФС устройств
/mnt     — mount — точки монтирования
/media   — съёмные носители (USB, CD)
/srv     — service — данные сервисов
/run     — данные запущенных процессов (современная замена /var/run)
/usr/bin     — бинарники пользовательских программ
/usr/lib     — библиотеки
/usr/local   — программы установленные локально (вне пакетного менеджера)
/var/log     — логи
/var/run     — pid файлы
/var/cache   — кэш
```

<a id="q-7-3"></a>
## 7.3. Структура записи в /etc/fstab

fstab — текстовый файл, список "что монтировать при загрузке". systemd читает его при старте.

```
/dev/sda1   /       ext4    defaults    0    1
↑           ↑       ↑       ↑           ↑    ↑
устройство  точка   тип ФС  опции      dump pass
            монт.
```

### Поля:
- **устройство** — `/dev/sda1`, `UUID=...`, `LABEL=...`
- **точка монтирования** — куда монтировать
- **тип ФС** — `ext4`, `xfs`, `tmpfs`, `swap`, `nfs`
- **опции** — `defaults`, `noatime`, `ro`, `noexec`
- **dump** — устаревшее, всегда `0`
- **pass** — порядок проверки `fsck`:
  ```
  0 — не проверять (swap, tmpfs, nfs)
  1 — первым (только корневая /)
  2 — после корневой (остальные разделы)
  ```

### Лучше использовать UUID:
```bash
blkid   # показать UUID всех устройств
```

UUID не меняется при добавлении нового диска — `/dev/sda` может измениться.

### Опции монтирования:
```
defaults  — стандартный набор (rw,suid,exec,auto,nouser,async)
ro        — read only
rw        — read write
noatime   — не обновлять atime
nodiratime— не обновлять atime для директорий
noexec    — нельзя запускать бинарники
nosuid    — игнорировать SUID биты
nodev     — игнорировать файлы устройств
sync      — синхронная запись (медленно но надёжно)
async     — асинхронная запись (дефолт)
auto      — монтировать при mount -a
noauto    — не монтировать при mount -a
user      — обычный пользователь может монтировать
```

<a id="q-7-4"></a>
## 7.4. Журналирование ext4 vs XFS

Журналирование — защита целостности ФС при внезапном отключении. Пишет в журнал что собирается сделать, при сбое восстанавливается из журнала.

### ext4 — три режима:
```
data=writeback  — только метаданные, данные могут быть старые после сбоя
data=ordered    — дефолт, данные пишутся до метаданных (без журнала)
data=journal    — всё журналируется, максимальная надёжность, медленнее
```

`data=journal` надёжнее потому что журналирует сами байты файла, не только метаданные. При `data=ordered` блоки данных могут быть записаны наполовину.

### XFS:
- Журналирует только метаданные (архитектурное решение, не ограничение)
- Лучше на параллельной записи — несколько потоков одновременно
- Лучше на больших файлах
- **Нельзя уменьшить** — только увеличить (`xfs_growfs`)
- Быстрее восстанавливается после сбоя
- Inode создаются динамически — нельзя исчерпать

### ext4 vs XFS — inode:
```bash
# ext4 — фиксированное количество inode при форматировании:
df -i   # можно исчерпать inode даже если место есть
touch newfile   # No space left on device — хотя df -h показывает место

# XFS — динамические inode, исчерпать практически невозможно
```

### Когда что:
```
ext4  — общего назначения, стабильная, можно уменьшить раздел
XFS   — большие файлы, высокая нагрузка записи, параллельность
```

Ubuntu/Debian — ext4 по умолчанию. RHEL/CentOS — XFS по умолчанию.

## 7.5. Перемонтировать корневую ФС в режим чтения-записи

```bash
mount -o remount,rw /
```

- `-o` — options, передать опции монтирования
- `remount` — перемонтировать с новыми опциями не размонтируя

Корневую ФС нельзя размонтировать пока система работает — `remount` единственный способ изменить опции на лету.

```bash
mount | grep "on / "   # проверить текущие опции
# → /dev/sda1 on / type ext4 (ro,relatime)
mount -o remount,rw /
# → (rw,relatime)
```

### Флаги mount:
```bash
-t ext4           # type — тип ФС
-o options        # options — опции
-a                # all — примонтировать всё из /etc/fstab
-r                # read only
-w                # read write
-l                # list — показать примонтированные ФС
```

## 7.6. Bind-монтирование

```bash
mount --bind /opt/nginx-1.25 /opt/nginx
```

Два пути к одним данным — оригинал никуда не девается:
```
/opt/nginx-1.25  ─┐
                   ├──→ одни и те же данные на диске
/opt/nginx       ─┘
```

Убрать bind-mount:
```bash
umount /opt/nginx   # оригинал /opt/nginx-1.25 жив
```

### Bind-mount vs симлинк:
```
симлинк   — файл с текстом пути, разыменовывается на уровне VFS
bind-mount — ядро регистрирует точку монтирования, прозрачно для всего
```

Симлинк не работает внутри chroot/контейнера — абсолютный путь цели не существует внутри изолированного окружения. Bind-mount работает — ядро уже знает куда это указывает до chroot.

### Подобие жёсткой ссылки для директорий:
```
жёсткая ссылка  — два имени одного inode (для файлов)
bind-mount       — два пути к одной директории на уровне ядра
```

### chroot:
```bash
chroot /jail /bin/bash   # / для этого bash = /jail
exit                      # вернуться обратно
```

Процесс не может выйти за пределы своего "корня". Внутри должны быть все нужные бинарники и библиотеки.

## 7.7. Монтирование раздела

```bash
# b — block device (первая буква в правах)
```

<a id="q-7-10"></a>
## 7.10. Swap-пространство

Когда RAM заканчивается — ядро выгружает давно неиспользуемые страницы памяти в swap. При обращении к ним — читает обратно в RAM (page fault → swap in). Процесс не знает что его память была в swap.

**Thrashing** — система непрерывно гоняет страницы между RAM и диском:
```bash
vmstat 1
# si — swap in (из swap в RAM)
# so — swap out (из RAM в swap)
# высокие si/so = thrashing
```

### Создать swap-файл 1GB:
```bash
# быстрый способ:
fallocate -l 1G /swapfile

# надёжный способ (dd пишет нули побайтово):
dd if=/dev/zero of=/swapfile bs=1M count=1024
# if = input file (/dev/zero — генерирует нули)
# of = output file
# bs = block size
# count = количество блоков (1024 × 1MB = 1GB)

chmod 600 /swapfile   # только root
mkswap /swapfile      # пометить как swap
swapon /swapfile      # включить

swapon --show         # проверить
free -h               # посмотреть память и swap
```

Постоянно — в `/etc/fstab`:
```
/swapfile  none  swap  sw  0  0
```

### /dev/zero vs /dev/null:
```
/dev/null  — поглощает всё (чёрная дыра)
/dev/zero  — генерирует нули (бесконечный источник)
```

## 8.1. MBR vs GPT

Таблица разделов — структура в начале диска которая описывает где какой раздел начинается и заканчивается. Каждый диск имеет свою таблицу.

```
диск /dev/sda:
[таблица разделов][раздел 1][раздел 2][раздел 3]
      ↑
   MBR или GPT
```

Зачем нужна — разбить один физический диск на независимые части:
```
/dev/sda (500GB):
├── /dev/sda1 (50GB)  → / (корневая ФС)
├── /dev/sda2 (400GB) → /home
└── /dev/sda3 (50GB)  → swap
```

Кто использует:
- BIOS/UEFI — при загрузке читает таблицу, находит загрузочный раздел
- Ядро — создаёт устройства `/dev/sda1`, `/dev/sda2`...
- `fdisk`, `parted` — утилиты редактирования

```bash
fdisk -l /dev/sda      # посмотреть таблицу разделов
parted /dev/sda print  # то же через parted
```

### MBR (Master Boot Record) — старый:
```
- максимум 4 первичных раздела
- максимальный размер диска — 2TB
- загрузчик в первых 512 байтах
- BIOS
```

### GPT (GUID Partition Table) — новый:
```
- до 128 разделов
- максимальный размер — 9.4 ZB
- резервная копия таблицы в конце диска
- UEFI
- каждый раздел имеет уникальный GUID
```

Сейчас везде GPT — MBR только на старом железе.

<a id="q-8-2"></a>
## 8.2. PV, VG, LV в LVM

**LVM** — Logical Volume Manager. Абстракция над физическими дисками для гибкого управления пространством. По сути балансер дискового пространства — позволяет перераспределять место между томами на лету.

### Три уровня:
```
физические диски
    ↓
PV (Physical Volume) — физический том (диск/раздел инициализированный под LVM)
    ↓
VG (Volume Group) — группа томов (общий пул пространства из всех PV)
    ↓
LV (Logical Volume) — логический том (виртуальный диск нарезанный из VG)
    ↓
файловая система (ext4, xfs...)
```

Аналогия:
```
PV — кирпичи (физические диски)
VG — склад (пул из всех кирпичей)
LV — комнаты (нарезаем из склада сколько нужно)
```

### LV — главная фишка:
Для системы выглядит как обычный раздел (`/dev/mapper/vg-lv`), но физически может быть размазан по нескольким дискам внутри VG. Система не знает об этом.

```
система видит:
/dev/mapper/storage-mysql  (150GB)  ← обычный "раздел"

реально:
100GB на /dev/sda
50GB  на /dev/sdb
```

### Пример — объединить два диска:
```bash
/dev/sda (500GB) → PV
/dev/sdb (500GB) → PV
                   ↓
            VG "storage" (1TB)
                   ↓
            LV "data" (1TB) → /data
```

### Пример — место кончается:
```bash
# /var заполнилась на 95%
pvcreate /dev/sdc              # новый диск → PV
vgextend vg /dev/sdc           # добавить в VG
lvextend -L +200G /dev/vg/var  # расширить LV
resize2fs /dev/vg/var          # расширить ФС
# система не останавливалась
```

### Когда нужен LVM:
- Продакшн серверы где нужна гибкость
- Когда заранее не знаешь сколько места нужно каждому разделу
- Когда нужны снапшоты

### Когда не нужен:
- Домашний ПК с одним диском
- Простой сервер где разделы никогда не меняются

LVM не балансирует сам — это инструмент в руках администратора.

<a id="q-8-3"></a>
## 8.3. Расширение LV и ФС

```bash
# 1. расширить логический том:
lvextend -L +50G /dev/vg/data       # добавить 50GB
lvextend -L 150G /dev/vg/data       # установить размер 150GB
lvextend -l +100%FREE /dev/vg/data  # использовать всё свободное место в VG

# 2. расширить файловую систему:
resize2fs /dev/vg/data              # для ext4
xfs_growfs /mnt/data                # для XFS (принимает точку монтирования)

# одной командой для ext4:
lvextend -L +50G --resizefs /dev/vg/data
```

### Почему сначала LV потом ФС:
ФС не знает о LVM — думает что работает с обычным блочным устройством определённого размера. Нельзя использовать пространство которое ФС не видит.

```
до:
LV  [==100GB==]
ФС  [==100GB==]

после lvextend:
LV  [==100GB==][+50GB свободно]
ФС  [==100GB==]  ← ФС не знает о новом месте

после resize2fs:
LV  [==100GB==][+50GB свободно]
ФС  [=========150GB===========] ← перестроилась
```

`resize2fs` читает суперблок, видит что устройство выросло, перестраивает внутренние структуры.

## 8.4. Striping и mirroring — RAID 10

### Striping (чередование) — RAID 0:
Данные делятся на части и пишутся на несколько дисков параллельно:
```
файл "hello world":
диск 1: "hello "
диск 2: "world"
```
✅ Скорость чтения/записи растёт  
❌ Один диск упал → все данные потеряны

### Mirroring (зеркалирование) — RAID 1:
Одни и те же данные пишутся на несколько дисков:
```
диск 1: "hello world"
диск 2: "hello world" (копия)
```
✅ Надёжность — один диск упал, данные живы  
❌ Полезный объём вдвое меньше

### RAID 10 — striping + mirroring:
```
4 диска:
[диск1][диск2] — зеркало
[диск3][диск4] — зеркало
     ↕               ↕
     stripe (данные чередуются между парами)
```
✅ И скорость и надёжность  
✅ Можно потерять один диск из каждой пары  
❌ Полезный объём = 50% от общего

<a id="q-8-8"></a>
## 8.8. Снапшоты LVM

Снапшот — мгновенный снимок состояния LV в определённый момент времени.

Механизм **COW (Copy-On-Write)**:
```
создали снапшот → снапшот указывает на те же блоки что и оригинал
кто-то изменил блок в оригинале → старое значение копируется в снапшот → потом блок меняется
```

Снапшот не копирует все данные сразу — только изменённые блоки. Поэтому создаётся мгновенно.

```bash
lvcreate -L 10G -s -n mysnap /dev/vg/data
# -s — snapshot
# -n — имя снапшота
# 10G — место для хранения изменений (COW область)
```

Сценарий использования:
```bash
# перед рискованным обновлением БД:
lvcreate -L 10G -s -n backup /dev/vg/mysql
# обновление сломалось → откатились:
lvconvert --merge /dev/vg/backup
```

## 8.9. RAID 1 vs RAID 5

| | RAID 1 | RAID 5 |
|--|--------|--------|
| Минимум дисков | 2 | 3 |
| Отказоустойчивость | 1 диск | 1 диск |
| Полезный объём | 50% | (N-1)/N |
| Скорость чтения | Высокая | Высокая |
| Скорость записи | Средняя | Ниже (штраф на parity) |
| Механизм | Зеркало | Чередование + parity |

RAID 5 — данные + контрольная сумма (parity) распределены по всем дискам:
```
3 диска по 100GB:
диск1: данные A
диск2: данные B  
диск3: parity(A XOR B)
полезный объём: 200GB (66%)
```

Один диск упал → восстанавливается из оставшихся данных и parity.

**Когда что:**
```
RAID 1 — простота, надёжность, небольшое количество дисков
RAID 5 — экономия места, больше дисков, высокая нагрузка чтения
```

## 8.10. Снапшот LVM — когда применять

Снапшот оправдан когда:
- Перед рискованной операцией (обновление БД, миграция)
- Для консистентного бэкапа работающей системы
- Для тестирования изменений с возможностью отката

```bash
# создать:
lvcreate -L 10G -s -n snap /dev/vg/data

# смонтировать снапшот (только для чтения):
mount -o ro /dev/vg/snap /mnt/snap

# откатиться:
lvconvert --merge /dev/vg/snap

# удалить снапшот:
lvremove /dev/vg/snap
```

Снапшот растёт по мере изменений в оригинале. Если COW область заполнилась — снапшот становится невалидным. Размер выбирать с запасом.

## 8.11. Создание LV — полная последовательность

```bash
# 1. Инициализировать физический том:
pvcreate /dev/sdb1
pvdisplay              # проверить

# 2. Создать группу томов:
vgcreate myvg /dev/sdb1
vgdisplay              # проверить

# 3. Создать логический том:
lvcreate -L 50G -n mydata myvg
lvdisplay              # проверить

# 4. Создать файловую систему:
mkfs.ext4 /dev/myvg/mydata

# 5. Примонтировать:
mkdir /mnt/data
mount /dev/myvg/mydata /mnt/data

# 6. Добавить в /etc/fstab для автомонтирования:
echo "/dev/myvg/mydata /mnt/data ext4 defaults 0 2" >> /etc/fstab
```

### Полезные команды LVM:
```bash
pvs    # краткий список PV
vgs    # краткий список VG
lvs    # краткий список LV

pvdisplay   # подробно о PV
vgdisplay   # подробно о VG
lvdisplay   # подробно о LV

vgextend myvg /dev/sdc    # добавить PV в VG
vgreduce myvg /dev/sdb1   # убрать PV из VG
lvremove /dev/myvg/mydata # удалить LV
```

## 9.2. Wants= vs Requires=

```ini

Requires=network.target   # жёсткая зависимость
```

- **`Wants=`** — запустить зависимость если возможно. Если зависимость не запустилась — сервис всё равно стартует
- **`Requires=`** — жёсткая зависимость. Если зависимость не запустилась — сервис тоже не запустится

```ini
# пример — nginx хочет сеть но не требует:
[Unit]
Wants=network-online.target
After=network-online.target   # запускаться ПОСЛЕ сети
```

`After=` — порядок запуска (не зависимость). `Before=` — наоборот.

<a id="q-9-4"></a>
## 9.4. Socket activation

Механизм при котором systemd слушает сокет вместо сервиса. Сервис запускается только когда приходит первое соединение.

```
обычно:
сервис запущен → слушает сокет → ждёт соединений (потребляет ресурсы вхолостую)

socket activation:
systemd слушает сокет → сервис не запущен → пришло соединение → systemd запускает сервис → передаёт соединение
```

Преимущества:
- Сервисы не запущены пока не нужны → меньше потребление памяти
- Параллельная загрузка — systemd открывает все сокеты сразу, сервисы стартуют параллельно
- Автоматический перезапуск — сокет остаётся открытым пока сервис перезапускается

Файлы:
```
nginx.socket   — описывает сокет
nginx.service  — описывает сервис
```

## 9.9. Targets vs уровни выполнения SysVinit

**SysVinit runlevels** — старая система:
```
0 — выключение
1 — однопользовательский (восстановление)
2 — многопользовательский без сети
3 — многопользовательский с сетью
5 — графический
6 — перезагрузка
```

**systemd targets** — современная система:
```
poweroff.target      ≈ runlevel 0
rescue.target        ≈ runlevel 1
multi-user.target    ≈ runlevel 3
graphical.target     ≈ runlevel 5
reboot.target        ≈ runlevel 6
```

Отличия:
- Target — просто группа юнитов, можно создать свои
- Несколько targets могут быть активны одновременно
- Зависимости между targets явные и гибкие
- Параллельный запуск юнитов внутри target

<a id="q-9-16"></a>
## 9.14. systemd и cgroups

systemd использует cgroups (control groups) для:
- Отслеживания всех процессов сервиса (включая форкнутые)
- Ограничения ресурсов (CPU, память, I/O)
- Гарантированного завершения всех процессов при остановке

```ini
[Service]
MemoryLimit=512M
CPUQuota=50%
```

При двойном форке (демонизация) — процесс выходит из-под контроля традиционных систем. systemd через cgroups отслеживает все процессы в группе независимо от форков.

При `systemctl stop` — systemd посылает SIGTERM всей cgroup, потом SIGKILL если не завершились. Ни один процесс не ускользнёт.

## 10.1. Низкоуровневые vs высокоуровневые инструменты

### Низкоуровневые — работают с конкретным файлом напрямую:
```bash
dpkg -i package.deb    # Debian/Ubuntu
rpm -i package.rpm     # RHEL/CentOS
```

Не знают о зависимостях — просто распаковывают и устанавливают.

### Высокоуровневые — работают с репозиториями:
```bash
apt install nginx      # Debian/Ubuntu
dnf install nginx      # RHEL/CentOS
```

Под капотом apt делает:
1. Идёт в репозиторий (список в `/etc/apt/sources.list`)
2. Смотрит индекс пакетов
3. Скачивает nginx + все зависимости
4. Устанавливает через dpkg

```
apt/dnf = менеджер который идёт в репо, берёт нужное и всё что к нему прилагается
dpkg/rpm = просто установщик конкретного файла
```

### Когда нужны низкоуровневые:
- Нет в репозитории (скачал .deb с сайта разработчика)
- Нет интернета (air-gapped сервер)
- Конкретная старая версия которой нет в репо
- Диагностика — посмотреть что внутри пакета

```bash
# если dpkg не установил зависимости:
dpkg -i myapp.deb      # упало
apt install -f         # -f = fix, доустановить зависимости

# перекинуть на другую машину без интернета:
apt download nginx     # скачать .deb без установки
scp nginx.deb user@machineB:/tmp/
dpkg -i /tmp/nginx.deb
```

### Полезные команды dpkg:
```bash
dpkg -l nginx          # статус пакета (ii=installed, rc=removed+configs, un=not installed)
dpkg -L nginx          # какие файлы установил пакет
dpkg -S /usr/bin/nginx # какому пакету принадлежит файл
```

## 10.2. Роль GPG ключей

GPG ключи защищают от подделки пакетов:
```
разработчик подписывает пакет приватным ключом
apt проверяет подпись публичным ключом
если не совпадает → отказывается устанавливать
```

Ключ выглядит как текстовый блок в base64:
```
-----BEGIN PGP PUBLIC KEY BLOCK-----
mQENBF6at6oBCAC6RUGH...
-----END PGP PUBLIC KEY BLOCK-----
```

Не каждое приложение — каждый **репозиторий** имеет свой ключ:
```
Ubuntu repo   → один ключ для всех пакетов
Docker repo   → свой ключ
Nginx repo    → свой ключ
```

### Добавить GPG ключ:
```bash
# современный способ:
curl -fsSL https://nginx.org/keys/nginx_signing.key | gpg --dearmor -o /etc/apt/trusted.gpg.d/nginx.gpg
```

Флаги curl:
```
-f   # fail — не выводить HTML ошибки
-s   # silent — без прогресса
-S   # Show error — показать ошибку если -s включён
-L   # Location — следовать редиректам
```

`gpg --dearmor` — конвертировать из текстового base64 формата в бинарный (apt хочет бинарный).

Ключи хранятся в `/etc/apt/trusted.gpg.d/`.

## 10.4. Закрепление (pinning/holding) пакетов


## 10.8. Локальный кэш пакетов

```
/var/cache/apt/archives/  — кэш .deb файлов (не критично если повреждён)
/var/lib/apt/lists/       — индексы репозиториев (критично если повреждён)
```

### Индексы — что это:
Текстовые файлы со списком всех пакетов в репозитории:
```
Package: nginx
Version: 1.24.0
Depends: libc6, libpcre3
Filename: pool/main/n/nginx/nginx_1.24.0_amd64.deb
SHA256: abc123...
```

`apt update` скачивает эти файлы. Без них apt не знает что вообще существует.

### Повреждение и починка:
```bash
# симптом:
apt install nginx
# E: Unable to locate package nginx

# починить:
rm -rf /var/lib/apt/lists/*
apt update
# или просто:
apt clean && apt update
```

```bash
apt clean       # удалить весь кэш .deb (освободить место)
apt autoclean   # удалить только устаревшие версии из кэша
```

Повреждается при: прерванном `apt update`, проблемах с диском, переполненном диске.

## Идея SUID — зачем вообще нужен

SUID решает одну конкретную проблему — как дать обычному пользователю выполнить действие которое требует привилегий, но только это конкретное действие.

**Проблема:**
```
/etc/shadow — только root может писать
обычный пользователь хочет сменить свой пароль
решение в лоб: дать всем root права → катастрофа
```

**Три участника:**
```
ядро      — проверяет EUID при доступе к файлу (тупо но быстро)
SUID бит  — подменяет EUID на владельца исполняемого файла
программа — проверяет бизнес-логику через RUID (чей пароль менять)
```

**Почему не другие решения:**
```
дать всем root       → слишком много прав
sudo для passwd      → нужно вводить пароль чтобы сменить пароль — абсурд
отдельный демон      → сложнее, но так делают в современных системах (polkit)
```

**Главная идея — минимальные привилегии:**
```
не давать пользователю root
не давать программе больше чем нужно
SUID даёт привилегии только конкретной программе
только на конкретное действие
только пока программа выполняется
```

SUID на скриптах не работает — только на скомпилированных ELF бинарниках. Скрипт интерпретируется другой программой — SUID сработал бы на интерпретаторе, это дыра.

## 11.1. Real, Effective, Saved UID

Три вида UID у каждого процесса:
```
RUID (real)      — кто реально запустил процесс (твой uid)
EUID (effective) — от чьего имени процесс работает прямо сейчас (ядро проверяет это)
SUID (saved)     — сохранённый, для возврата привилегий
```

Аналогия:
```
RUID = паспорт (кто ты на самом деле)
EUID = бейджик (какие права у тебя сейчас)
```

Ядро проверяет EUID при каждом обращении к файлу — не RUID.

### Пример с passwd:
```bash
ls -la /usr/bin/passwd
# -rwsr-xr-x root root /usr/bin/passwd  ← SUID бит

# ты (uid=1000) запускаешь passwd:
RUID = 1000   # ты
EUID = 0      # root (из-за SUID бита)
# passwd может писать в /etc/shadow потому что EUID=root
# но passwd проверяет RUID — позволяет менять только твой пароль
```

### Saved UID:
Процесс может временно сбросить привилегии и вернуться:
```
EUID=0 → сбросить до EUID=1000 → поработать без привилегий → вернуть EUID=0 (из saved)
```

<a id="q-11-3"></a>
## 11.2. Права rwx для файлов и директорий

```bash
ls -la
# -rwxr-xr-- 1 god staff file.txt
# drwxr-x--- 2 god staff mydir/
```

Три роли всегда в порядке: **владелец → группа → остальные**

### Для файлов:
```
r — читать содержимое
w — изменять содержимое
x — выполнять как программу
```

### Для директорий — другая логика:
```
r — читать список файлов (ls)
w — создавать/удалять файлы внутри
x — входить (cd) и обращаться к файлам внутри
```

`x` на директории критичен — без него вообще ничего не можешь делать внутри:
```bash
chmod 600 /mydir   # убрали x
cd /mydir          # → Permission denied
ls /mydir          # → Permission denied даже с r!
```

Интересная комбинация:
```bash
chmod 111 /mydir   # только x, без r
cd /mydir          # можно войти
ls /mydir          # нельзя (нет r)
cat /mydir/file    # можно если знаешь имя файла!
```

## 11.3. Добавить пользователя в группу

```bash
usermod -aG groupname username
```

## 11.4. Sticky Bit

```bash
ls -la /tmp
# drwxrwxrwt   ← t = sticky bit
```

Sticky bit на директории — удалять файлы может только владелец файла, владелец директории, или root:

```bash
chmod +t /shared      # установить
chmod 1777 /shared    # то же числом (1 = sticky)
```

```
drwxrwxrwt — t = sticky + x есть
drwxrwxrwT — T = sticky, x нет
```

Важно: sticky не запрещает редактировать чужие файлы — только удалять. Редактирование контролируется правами самого файла.

```
права директории → создание/удаление файлов внутри
права файла      → чтение/запись самого файла
```

## 11.5. ACL — Access Control List

Расширенные права — позволяют задать права конкретному пользователю/группе помимо стандартных трёх ролей.

```bash
getfacl file.txt
# user::rw-
# user:vasya:r--    ← конкретный пользователь
# group::r--
# mask::r--
# other::r--
```

### Синтаксис ACL записи:
```
u:vasya:rw   # user — конкретный пользователь
g:dev:rx     # group — конкретная группа
o::r         # other — все остальные
m::rw        # mask — потолок прав для ACL записей
u::rw        # владелец файла (без имени)
g::r         # группа файла (без имени)
```

### setfacl флаги:
```bash
-m   # modify — добавить/изменить
-x   # remove — удалить запись
-b   # blank — удалить все ACL
-k   # удалить только default ACL
-d   # default ACL (для директорий, наследуется новыми файлами)
-R   # recursive
```

```bash
setfacl -m u:vasya:rw file.txt        # дать vasya права rw
setfacl -m g:dev:rx file.txt          # дать группе dev права rx
setfacl -x u:vasya file.txt           # убрать ACL для vasya
setfacl -b file.txt                   # убрать все ACL
setfacl -Rm u:vasya:rw /shared        # рекурсивно
setfacl -d -m u:vasya:rw /shared      # default ACL для новых файлов
```

### Когда нужны ACL:
Стандартные права — только три роли. ACL нужен когда:
- Разные пользователи имеют разные права
- Нельзя решить через группы
- Нужны права для конкретного пользователя без добавления в группу

## 11.6. PAM — Pluggable Authentication Modules

Модульная система аутентификации. Программы не знают как аутентифицировать — спрашивают PAM.

```
ssh → PAM → /etc/pam.d/sshd → список модулей
```

Каждый сервис имеет свой конфиг в `/etc/pam.d/`:
```
/etc/pam.d/sshd    — SSH
/etc/pam.d/sudo    — sudo
/etc/pam.d/login   — консоль
/etc/pam.d/passwd  — смена пароля
```

### Четыре типа модулей:
```
auth     — аутентификация (проверка пароля)
account  — проверка аккаунта (не истёк ли)
password — смена пароля
session  — действия при входе/выходе
```

### Флаги модулей:
```
required   — должен пройти, но продолжает остальные (для timing атак)
requisite  — провал → сразу стоп
sufficient — прошёл → достаточно, остальные не нужны
optional   — необязательный
```

`required` вместо `requisite` для безопасности — одинаковое время ответа при любой ошибке, сложнее определить причину отказа. Плюс все модули выполняются (например pam_tally2 считает попытки).

Добавить двухфакторку без изменения SSH:
```bash
# /etc/pam.d/sshd
auth required pam_unix.so
auth required pam_google_authenticator.so
```

<a id="q-11-8"></a>
## 11.7. umask

Маска вычитается из максимальных прав при создании файла:

```
максимум файла:      666 (без x — файлы не исполняемые по умолчанию)
максимум директории: 777

umask 027:
файл:      666 - 027 = 640 (rw-r-----)
директория: 777 - 027 = 750 (rwxr-x---)

дефолт umask 022:
файл:      666 - 022 = 644 (rw-r--r--)
директория: 777 - 022 = 755 (rwxr-xr-x)
```

Почему файлы максимум 666 (без x) — файлы по умолчанию не должны быть исполняемыми. `x` добавляется только явно через `chmod +x`.

```bash
umask              # посмотреть текущий
umask 027          # установить для сессии
# постоянно — в ~/.bashrc или /etc/profile
```

## 11.8. ulimit — ограничения ресурсов

```bash
ulimit -a          # все лимиты
ulimit -n 1024     # открытые файлы
ulimit -u 100      # процессы
ulimit -Sn 1024    # soft лимит
ulimit -Hn 4096    # hard лимит
```

### Soft vs Hard:
```
soft — текущий лимит, пользователь может увеличить до hard
hard — потолок, только root может увеличить
```

Пользователь: soft → увеличить до hard ✅, hard → только уменьшить.

Постоянно в `/etc/security/limits.conf`:
```
god  soft  nofile  1024
god  hard  nofile  4096
*    soft  nproc   100    # для всех
```

Ядро проверяет лимиты при каждом syscall (`open()`, `fork()`).

## 11.9. SUID, SGID, Sticky bit

```
SUID (4) — на файле: EUID = владелец файла при запуске
SGID (2) — на файле: EGID = группа файла при запуске
           на директории: новые файлы наследуют группу директории
Sticky(1) — на директории: удалять может только владелец файла
```

```bash
chmod u+s file    # SUID (u = user/владелец)
chmod g+s file    # SGID (g = group)
chmod +t dir      # sticky

# числами (первая цифра):
chmod 4755 file   # SUID
chmod 2755 dir    # SGID
chmod 1777 dir    # sticky
```

В `ls`:
```
-rwsr-xr-x   # SUID — s вместо x у владельца
-rwxr-sr-x   # SGID — s вместо x у группы
drwxrwxrwt   # sticky — t вместо x у остальных
-rwSr-xr-x   # SUID есть, но x нет (S заглавная — редко, обычно баг)
```

### SUID на скриптах не работает:
Ядро игнорирует SUID на скриптах (bash, python) — только на скомпилированных ELF бинарниках. Скрипт интерпретируется другой программой — SUID сработал бы на интерпретаторе, это дыра в безопасности.

### Пример SUID утилиты:
```bash
ls -la /usr/bin/passwd
# -rwsr-xr-x root root /usr/bin/passwd
```

### Принцип минимальных привилегий:
SUID даёт повышенные права только конкретной программе на конкретное действие — не всем пользователям root права.

## 11.10. Блокировка учётной записи

```bash
usermod -L god        # Lock — заблокировать
usermod -U god        # Unlock — разблокировать
passwd -l god         # то же через passwd
passwd -u god         # разблокировать
```

В `/etc/shadow` добавляется `!` перед хэшем:
```
god:$6$abc123$хэш:...    # разблокирован
god:!$6$abc123$хэш:...   # заблокирован
```

**Важно:** вход по SSH ключу работает даже при `-L` — ключи не через пароль!

Полная блокировка:
```bash
usermod -L god                    # заблокировать пароль
usermod -s /sbin/nologin god      # запретить интерактивный вход
```

`/sbin/nologin` — не shell, просто выводит "account not available" и выходит.

Но даже с `/sbin/nologin` можно выполнить команду:
```bash
ssh god@server "ls /home/god"   # без интерактивного входа работает
```

## 11.12. sudoers — конкретная команда без пароля

```bash
visudo   # безопасный редактор /etc/sudoers (проверяет синтаксис)
visudo -f /etc/sudoers.d/god   # отдельный файл
```

```bash
sudo -l   # посмотреть что можешь делать
```

### /etc/sudoers.d/ — директория фрагментов:
Вместо одного большого файла — много маленьких:
```
/etc/sudoers.d/
├── god
├── developers
└── docker
```

В `/etc/sudoers` есть `#includedir /etc/sudoers.d` — sudo читает все файлы из директории.

Паттерн `.d` (directory) везде в Linux:
```
/etc/sudoers.d/
/etc/cron.d/
/etc/apt/sources.list.d/
/etc/nginx/conf.d/
```

Идея — вместо одного большого файла, много маленьких фрагментов.

## /etc/passwd и /etc/shadow

```bash
cat /etc/passwd
# god:x:1000:1000::/home/god:/bin/bash
# имя:пароль:uid:gid:комментарий:home:shell
# x = пароль в /etc/shadow
```

```bash
cat /etc/shadow
# god:$6$abc123$хэш:19000:0:99999:7:::
# имя:хэш_пароля:дата_смены:...
```

`/etc/passwd` — читают все (нужен для uid→имя).

Раньше хэши были в `/etc/passwd` — любой мог брутфорсить оффлайн. Вынесли в shadow.

## 12.1. AppArmor vs SELinux

Оба — MAC (Mandatory Access Control). Работают по-разному:

### AppArmor:
- Политики основаны на **путях к файлам**
- Проще в настройке
- Профиль для каждой программы
- Дефолт на Ubuntu/Debian

```bash
# профиль живёт в:
/etc/apparmor.d/usr.sbin.nginx

# nginx может:
/etc/nginx/** r,         # читать конфиги
/var/log/nginx/** w,     # писать логи
# nginx не может:
# /etc/shadow — не прописано → запрещено
```

### SELinux:
- Политики основаны на **контекстах безопасности** (labels)
- Сложнее но мощнее
- Дефолт на RHEL/CentOS

```bash
ls -Z /etc/shadow
# system_u:object_r:shadow_t:s0 /etc/shadow
```

### Главное различие:
```
AppArmor — путь к файлу (/etc/shadow)
SELinux  — контекст файла (shadow_t) — не зависит от пути
```

Переместил файл — AppArmor потеряет (путь изменился). SELinux не потеряет (контекст остался).

<a id="q-12-2"></a>
## 12.2. DAC vs MAC

**DAC (Discretionary Access Control)** — дискреционный:
```
владелец сам решает кому давать доступ
chmod, chown, ACL — всё это DAC
```

Проблема:
```
скомпрометированный nginx работает как root
→ читает /etc/shadow (DAC разрешает — root же)
→ данные уже утекли до того как заметили
```

**MAC (Mandatory Access Control)** — мандатный:
```
система устанавливает правила, владелец не может их изменить
SELinux, AppArmor — MAC
```

```
nginx скомпрометирован
→ пытается читать /etc/shadow
→ SELinux: httpd_t не может читать shadow_t → отказ
→ данные не утекли
```

Два уровня защиты:
```
DAC (unix права) → MAC (SELinux/AppArmor) → доступ
оба должны разрешить
```

MAC не про то что root бессилен — про **least privilege**: nginx не должен иметь доступ к /etc/shadow вообще, зачем давать?

## 12.3. auditd — классы событий

```bash
# добавить правило:
auditctl -w /etc/passwd -p rwxa -k passwd_changes
# -w = watch (файл)
# -p = permissions (r=read, w=write, x=execute, a=attribute)
# -k = key (метка для поиска)

# посмотреть логи:
ausearch -k passwd_changes
ausearch -m avc -ts recent   # SELinux события
aureport --summary            # сводный отчёт
```

Логи в `/var/log/audit/audit.log`.

Классы событий:
```
syscall  — системные вызовы
file     — доступ к файлам
user     — логин, sudo
network  — соединения
process  — создание/завершение
```

Для сетевых соединений удобнее:
```bash
ss -tnp      # TCP соединения с процессами
lsof -i      # все сетевые соединения
tcpdump      # перехват пакетов
```

Заблокировать изменение правил (immutable mode):
```bash
auditctl -e 2   # даже root не может остановить без перезагрузки
```

<a id="q-12-4"></a>
## 12.4. auditd vs стандартное журналирование

```
journald/syslog:
- приложения сами решают что логировать
- можно пропустить важные события
- процесс может скрыть свои действия

auditd:
- работает на уровне ядра — перехватывает syscall-ы
- процесс не может скрыть действия
- tamper-resistant
- кто/что/когда с точностью до пользователя
```

```bash
# journald — что приложение решило записать:
journalctl -u nginx
# → "nginx started", "GET /index.html 200"

# auditd — что реально происходило на уровне ядра:
ausearch -f /etc/shadow
# → кто открывал, когда, какой процесс, какой uid
```

### Как обойти auditd:
- Обычный процесс — не может
- Root — может выключить, но это залогируется
- Kernel rootkit — может обойти (работает до auditd)

**Rootkit** — вредоносный модуль ядра который подменяет syscall table:
```c
// подменить sys_open своей функцией:
syscall_table[__NR_open] = evil_open;

// evil_open скрывает нужные файлы от auditd
```

Защита — логи на отдельный сервер в реальном времени + `auditctl -e 2`.

Обнаружение rootkit:
```bash
rkhunter --check    # rootkit hunter
chkrootkit          # проверка известных rootkit-ов
# надёжнее — загрузиться с внешнего носителя
```

## 12.5. Четыре компонента контекста SELinux (*)

```
system_u : object_r : shadow_t : s0
   ↑           ↑         ↑       ↑
 юзер        роль      тип    уровень
```

- **user** — SELinux пользователь. `system_u`, `unconfined_u`
- **role** — роль. `object_r` для файлов, `system_r` для процессов
- **type** — **самое важное**. Определяет политику доступа
- **level** — уровень секретности (MLS). Обычно `s0`

На практике смотришь только на **type**.

Политика = огромная матрица:
```
кто (тип процесса) × что (тип файла) × действие = разрешено/запрещено

httpd_t → shadow_t → read = ЗАПРЕЩЕНО
httpd_t → httpd_sys_content_t → read = РАЗРЕШЕНО
passwd_t → shadow_t → write = РАЗРЕШЕНО
```

## 13.2. Маршрут по умолчанию vs статический маршрут

```bash
ip route show
# default via 192.168.1.1 dev eth0        ← маршрут по умолчанию
# 192.168.1.0/24 dev eth0 proto kernel    ← локальная сеть
# 10.0.0.0/8 via 10.0.0.1 dev eth1       ← статический маршрут
```

**Default gateway** — куда отправлять если не знаешь куда. Обычно выход в интернет.

**Статический маршрут** — конкретная подсеть через конкретный шлюз:
```
офисная сеть:
- интернет → через 192.168.1.1 (провайдер) ← default
- корпоративная 10.0.0.0/8 → через 10.0.0.1 (VPN) ← статический
```

Как ядро выбирает:
```
самый специфичный маршрут выигрывает:
/24 > /8 > default
```

Пакет прыгает от роутера к роутеру — каждый знает только следующий шаг (эстафета). Статический маршрут = "знаю через какой шлюз идти для этой подсети", не весь путь.

<a id="q-13-3"></a>
## 13.3. iptables vs nftables

### iptables — старый:
```
- цепочки: INPUT, OUTPUT, FORWARD
- таблицы: filter, nat, mangle
- правила проверяются линейно сверху вниз O(n)
- нет атомарного обновления
```

### nftables — новый:
```
- один инструмент вместо iptables/ip6tables/arptables
- атомарное обновление — все правила за раз
- sets — хранить IP/порты в множествах, проверка O(1)
- быстрее
```

### Sets в nftables:
```bash
# iptables — правило для каждого IP:
iptables -A INPUT -s 1.1.1.1 -j DROP
iptables -A INPUT -s 2.2.2.2 -j DROP

# nftables — один set:
nft add set ip filter blacklist { type ipv4_addr\; }
nft add element ip filter blacklist { 1.1.1.1, 2.2.2.2 }
nft add rule ip filter INPUT ip saddr @blacklist drop
```

### Где применяются:
- На сервере — фильтрация входящего трафика
- На роутере/шлюзе — NAT, фильтрация
- Docker — автоматически создаёт правила iptables
- Kubernetes — управление трафиком между подами

<a id="q-13-4"></a>
## 13.4. Правила iptables — SSH разрешить, остальное запретить

```bash
# разрешить уже установленные соединения:
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# разрешить loopback:
iptables -A INPUT -i lo -j ACCEPT

# разрешить SSH:
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# запретить всё остальное:
iptables -A INPUT -j DROP
```

### Синтаксис:
```
-A INPUT     # Append в цепочку INPUT (входящие)
-p tcp       # protocol — TCP
--dport 22   # destination port
-j ACCEPT    # jump — принять
-j DROP      # отбросить (без уведомления)
-j REJECT    # отбросить + уведомить отправителя
-i lo        # input interface — loopback
-m state     # match state — модуль состояния
```

### Почему TCP для SSH:
SSH работает поверх TCP — нужна надёжная доставка и порядок пакетов. UDP без гарантий.

### Почему loopback:
`lo` (127.0.0.1) — трафик между процессами на одной машине. Без него nginx не достучится до php-fpm, приложение до mysql.

### Почему ESTABLISHED,RELATED:
Без этого ответы на исходящие запросы блокируются:
```
curl google.com → исходящий разрешён
google → ответ → INPUT → DROP без ESTABLISHED!
```

```bash
iptables -L -n -v   # посмотреть правила
iptables -F         # очистить все правила
```

<a id="q-13-6"></a>
## 13.6. Сетевой мост (L2) vs маршрутизатор (L3)

### Стек TCP/IP:
```
L4 — транспортный  (TCP, UDP) — порты, надёжность
L3 — сетевой       (IP)       — адресация, маршрутизация
L2 — канальный     (Ethernet) — MAC адреса, одна сеть
L1 — физический    (кабель)   — биты, сигналы
```

**Мост (L2):**
```
видит MAC адреса, не знает про IP
все устройства в одной подсети
[192.168.1.1] ─┐
                ├── bridge ── [192.168.1.3]
[192.168.1.2] ─┘
```

**Маршрутизатор (L3):**
```
видит IP адреса
соединяет разные подсети
[192.168.1.0/24] ── router ── [10.0.0.0/8]
```

Чем выше уровень — тем больше информации для фильтрации:
```
L2 — только MAC (легко подделать)
L3 — IP адреса, подсети
L4 — порты, состояние TCP соединения → iptables работает здесь
```

## 13.8. IP-маскарадинг (NAT)

Приватные адреса (`192.168.x.x`, `10.x.x.x`) — интернет их не знает, ответ потеряется.

Маскарадинг — подмена исходящего IP на адрес шлюза:
```
ПК (192.168.1.5) → роутер подменяет на 85.123.45.67 → google
google → 85.123.45.67 → роутер знает что это был 192.168.1.5 → возвращает
```

```bash
# включить маскарадинг:
iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE
# -t nat      — таблица NAT
# POSTROUTING — после принятия решения о маршруте
# -o eth0     — исходящий интерфейс
# MASQUERADE  — подменить на IP интерфейса
```

Зачем нужно:
- Экономия IPv4 адресов — весь дом/офис через один внешний IP
- Приватные адреса не маршрутизируются в интернете

Docker делает то же — контейнеры выходят в интернет через IP хоста.

## 14.1. Симметричное vs асимметричное шифрование

### Симметричное:
```
один ключ для шифрования и расшифровки
проблема — как безопасно передать ключ?
плюс — быстрое
```

### Асимметричное:
```
пара математически связанных ключей:
приватный — только у тебя, никому не давать
публичный — можно всем раздавать

зашифровано публичным → расшифровать только приватным
подписано приватным → проверить только публичным
```

Зная публичный ключ — нельзя восстановить приватный (задача факторизации больших чисел).

### Как SSH использует оба:
```
1. Подключение → асимметричное (обмен Диффи-Хеллмана)
   → генерируется временный симметричный ключ сессии

2. Весь трафик → симметричное (AES) — быстрее

3. Аутентификация → асимметричное
   → сервер проверяет твой приватный ключ через публичный
```

## 14.3. ssh-agent

Хранит приватные ключи в памяти в расшифрованном виде:

## 14.4 и 14.10. Проброс портов (туннели)

```bash
# Local (-L) — локальный порт → удалённый сервис:
ssh -L 5432:db.internal:5432 user@bastion
# localhost:5432 → bastion → db.internal:5432

# Remote (-R) — удалённый порт → локальный сервис:
ssh -R 8080:localhost:3000 user@server

```

```bash
ssh -L 8080:remote_host:80 user@bastion
# или через ProxyJump:
ssh -J bastion -L 8080:remote_host:80 user@remote_host
```

<a id="q-14-5"></a>
## 14.5. known_hosts

```bash
~/.ssh/known_hosts   # публичные ключи известных серверов
```

При первом подключении — SSH спрашивает подтвердить fingerprint сервера → сохраняет в known_hosts.

При следующих подключениях — проверяет ключ сервера:
```
совпадает → всё ок
изменился → WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!
```

Изменившийся ключ = переустановили сервер (нормально) или MITM атака (опасно).

```bash
ssh-keygen -R server.com   # удалить запись (после переустановки)
```

### Защита от MITM:
```
без known_hosts:
ты → злоумышленник (притворяется сервером) → видит весь трафик

с known_hosts:
ключ изменился → WARNING → проверить перед вводом пароля
```

## 14.6. Клиентский блок конфигурации

```bash
# ~/.ssh/config:
Host myserver
    HostName server.example.com
    Port 2222
    User god
    IdentityFile ~/.ssh/id_rsa_server

Host bastion
    HostName 1.2.3.4
    User admin
    IdentityFile ~/.ssh/bastion_key

Host internal
    HostName 10.0.0.5
    ProxyJump bastion         # подключиться через bastion

Host *
    ServerAliveInterval 60    # keepalive каждые 60 секунд
    ServerAliveCountMax 3
```

```bash
ssh myserver   # вместо: ssh -p 2222 -i ~/.ssh/id_rsa_server god@server.example.com
```

## 14.7. Риски agent forwarding

```bash
ssh -A user@server   # включить agent forwarding
```

Твой агент доступен на удалённом сервере — можно подключаться дальше без пароля.

**Риск:**
```

приватный ключ не виден, но агент можно использовать
```

Безопаснее — `ProxyJump`:

```bash
ssh-copy-id user@server
ssh-copy-id -i ~/.ssh/id_rsa.pub user@server

# вручную:
cat ~/.ssh/id_rsa.pub | ssh user@server "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
```

Права которые должны быть:
```
~/.ssh/              — 700
~/.ssh/authorized_keys — 600
```

SSH откажется работать если права слишком открытые.

## 15.1. Cron vs таймеры systemd

### Cron — классический планировщик:
```bash
crontab -e   # редактировать задачи текущего пользователя
crontab -l   # посмотреть задачи

# формат строки cron:
# минута час день_месяца месяц день_недели команда
  0      3   *           *     *           /usr/local/bin/backup.sh
# каждый день в 3:00

# спецсимволы:
# * = любое значение
# */5 = каждые 5 единиц
# 1,3,5 = конкретные значения
# 1-5 = диапазон
```

### Таймеры systemd:
```bash
systemctl list-timers          # список всех таймеров
systemctl status backup.timer  # статус конкретного
```

### Сравнение:

| | Cron | systemd timer |
|--|------|--------------|
| Точность | минута | секунда и меньше |
| Логирование | отдельно (mail/syslog) | journald автоматически |
| Зависимости | нет | полные systemd зависимости |
| Пропущенные задачи | теряются | Persistent=true догонит |
| Сложность | проще | сложнее |

Cron проще для простых задач. systemd timer — когда нужна интеграция с systemd, зависимости, точное логирование.

<a id="q-15-2"></a>
## 15.2. Монотонное vs календарное планирование в systemd

### Календарное (realtime) — привязано к реальному времени:
```ini
[Timer]
OnCalendar=*-*-* 03:00:00    # каждый день в 3:00
OnCalendar=Mon *-*-* 00:00   # каждый понедельник
OnCalendar=weekly            # раз в неделю
OnCalendar=daily             # раз в день
```

### Монотонное — привязано к событию:
```ini
[Timer]
OnBootSec=5min          # через 5 минут после загрузки
OnUnitActiveSec=1h      # каждый час после последнего запуска
OnStartupSec=10min      # через 10 минут после старта systemd
```

**Монотонное** — не зависит от системного времени. Если изменить время — не повлияет.
**Календарное** — привязано к часам. Изменили время → задача запустится в нужный момент.

Реальный сценарий:
```
резервное копирование в 3:00 → OnCalendar (нужно конкретное время)
проверка после загрузки → OnBootSec (нужно после события)
```

## 15.3. anacron vs cron

**Проблема cron:** если машина была выключена в момент запуска задачи — задача пропускается навсегда.

```
cron: backup каждый день в 3:00
машина выключена в 3:00 → backup не выполнился → потеряли бэкап
```

**anacron** решает это:
```
anacron проверяет при загрузке — когда последний раз выполнялась задача?
если давно — выполняет сразу
```

```bash
# /etc/anacrontab:
# период  задержка  имя_задачи  команда
  1       5         backup      /usr/local/bin/backup.sh
# период 1 = ежедневно
# задержка 5 = запустить через 5 минут после загрузки
```

anacron не умеет запускать задачи в конкретное время — только с периодичностью (ежедневно, еженедельно). Для серверов которые работают 24/7 — cron. Для ноутбуков/десктопов — anacron.

## 15.4. journald — RAM vs диск

**journald** — демон журналирования systemd. Собирает логи со всех сервисов автоматически.

### Хранение по умолчанию — только в RAM:
```
/run/log/journal/   — volatile (энергозависимое), в RAM
при перезагрузке → логи исчезают
```

### Включить постоянное хранение:
```bash
mkdir -p /var/log/journal
systemd-tmpfiles --create --prefix /var/log/journal
systemctl restart systemd-journald
```

Или в конфиге:
```bash
# /etc/systemd/journald.conf:
Storage=persistent    # всегда на диск
Storage=volatile      # всегда в RAM
Storage=auto          # на диск если /var/log/journal существует (дефолт)
```

```bash
journalctl --disk-usage   # сколько места занимает журнал
journalctl --vacuum-size=500M   # оставить только 500MB
journalctl --vacuum-time=2weeks # оставить только за 2 недели
```

## 15.6. logrotate

**logrotate** — утилита ротации логов. Предотвращает бесконечный рост лог файлов.

```bash
# /etc/logrotate.d/nginx:
/var/log/nginx/*.log {
    daily           # ротировать ежедневно
    missingok       # не ошибка если файл отсутствует
    rotate 14       # хранить 14 старых файлов
    compress        # сжимать старые логи (gzip)
    delaycompress   # не сжимать последний ротированный
    notifempty      # не ротировать если файл пустой
    create 0640 nginx adm  # создать новый файл с этими правами
    sharedscripts   # выполнить скрипты один раз для всех файлов
    postrotate
        nginx -s reopen   # сказать nginx открыть новые файлы
    endscript
}
```

### postrotate:
Блок команд которые выполняются ПОСЛЕ ротации. Нужен потому что:
```
logrotate переименовал nginx.log → nginx.log.1
создал новый пустой nginx.log
но nginx всё ещё пишет в старый файл (держит fd открытым)
postrotate: nginx -s reopen → nginx открывает новый nginx.log
```

Без postrotate — nginx продолжал бы писать в переименованный файл.

```bash
logrotate -f /etc/logrotate.d/nginx   # принудительно запустить
logrotate -d /etc/logrotate.d/nginx   # dry-run (симуляция)
```


## 15.9. journald — почему логи теряются после перезагрузки

По умолчанию `Storage=auto` — journald пишет в RAM (`/run/log/journal/`).

`/run` — tmpfs (в памяти), очищается при перезагрузке. Поэтому логи исчезают.

Если существует `/var/log/journal/` — journald автоматически начинает писать туда (на диск). Постоянное хранение.

```bash
# проверить текущий режим:
journalctl --header | grep "File Path"

# логи прошлых загрузок (если persistent):
journalctl -b -1   # предыдущая загрузка
journalctl -b -2   # позапрошлая
journalctl --list-boots  # список всех загрузок
```

## 16.2. ENTRYPOINT vs CMD в Dockerfile

```dockerfile
FROM ubuntu

ENTRYPOINT ["/usr/bin/nginx"]   # основная команда, не перезаписывается
CMD ["-g", "daemon off;"]       # аргументы по умолчанию, перезаписываются
```

### Как взаимодействуют:
```bash
# дефолтный запуск:
docker run myimage
# → /usr/bin/nginx -g "daemon off;"

# переопределить CMD:
docker run myimage -c /etc/nginx/custom.conf
# → /usr/bin/nginx -c /etc/nginx/custom.conf

# переопределить ENTRYPOINT:
docker run --entrypoint /bin/bash myimage
# → /bin/bash
```

### Форматы:
```dockerfile
# exec форма (рекомендуется) — прямой запуск, PID 1:
ENTRYPOINT ["/usr/bin/nginx"]
CMD ["-g", "daemon off;"]

# shell форма — запускается через /bin/sh -c:
ENTRYPOINT nginx
CMD -g daemon off;
# проблема: сигналы (SIGTERM) не доходят до процесса — идут в sh
```

**Правило:** всегда используй exec форму (массив) — сигналы доходят правильно.

## 16.4. Именованные тома vs bind-монтирования

```
bind mount   → разработка, конфиги, логи доступные с хоста
```

<a id="q-16-6"></a>
## 16.6. PID namespace и проблема PID 1

### PID namespace:
Каждый контейнер имеет своё пространство PID — процессы контейнера не видят процессы хоста.

```
хост:  PID 1=systemd, PID 100=nginx, PID 200=docker
контейнер: PID 1=nginx (внутри контейнера)
           хостовый PID nginx может быть 5678
```

### Проблема PID 1 в контейнере:
PID 1 в Linux имеет особую роль — усыновляет осиротевшие процессы и обрабатывает сигналы.

Обычный процесс как PID 1 в контейнере:
```
docker stop контейнер → SIGTERM → nginx (PID 1)
nginx не обрабатывает SIGTERM как PID 1 → 10 секунд → SIGKILL
```

Решения:
```dockerfile
# 1. exec форма в Dockerfile (процесс становится PID 1 напрямую):
ENTRYPOINT ["/usr/bin/nginx", "-g", "daemon off;"]

# 2. tini — минимальный init для контейнеров:
RUN apt install -y tini
ENTRYPOINT ["/usr/bin/tini", "--", "/usr/bin/nginx"]

# 3. --init флаг:
docker run --init nginx
```

## 16.10. Шесть типов namespaces Linux

Namespaces — механизм изоляции процессов в ядре Linux. Docker использует все шесть:

```
PID     — изоляция процессов (свои PID, не видят хост)
Network — изоляция сети (свои интерфейсы, таблица маршрутизации)
Mount   — изоляция файловой системы (своё дерево директорий)
UTS     — изоляция hostname и domainname (своё имя хоста)
IPC     — изоляция межпроцессного взаимодействия (shared memory, очереди)
User    — изоляция пользователей (root внутри = не root снаружи)
```

```bash
# посмотреть namespaces процесса:
ls -la /proc/1234/ns/
```

cgroups (control groups) — дополняют namespaces: не изолируют, а **ограничивают** ресурсы (CPU, память, I/O).

```
namespaces = что процесс ВИДИТ
cgroups    = сколько ресурсов процесс ПОЛУЧАЕТ
```

## 16.12. User namespaces

Без user namespaces:
```
root внутри контейнера = root на хосте
прорыв из контейнера → root на хосте
```

С user namespaces:
```
UID 0 (root) внутри → UID 65534 (nobody) на хосте
прорыв из контейнера → непривилегированный пользователь на хосте
```

```bash
# включить в Docker:
# /etc/docker/daemon.json:
{
  "userns-remap": "default"
}

systemctl restart docker
```

После включения Docker создаёт пользователя `dockremap` и маппит UID:
```
внутри контейнера: UID 0-65535
на хосте: UID 100000-165535
```

## 16.14. Многоэтапный Dockerfile (*)

```dockerfile
# Этап 1 — сборка:
FROM node:18 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
RUN npm run build

# Этап 2 — продакшн образ:
FROM node:18-alpine
WORKDIR /app

# копируем только собранное из этапа сборки:
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules

# непривилегированный пользователь:
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# корректная пересылка сигналов:
RUN apk add --no-cache tini
ENTRYPOINT ["/sbin/tini", "--"]

CMD ["node", "dist/index.js"]
```

Плюсы многоэтапной сборки:
```
финальный образ не содержит:
- инструменты сборки (компилятор, npm dev зависимости)
- исходный код
- временные файлы
→ меньше размер, меньше attack surface
```

### Почему tini для сигналов:
```
docker stop → SIGTERM → PID 1 (node)
node не всегда правильно обрабатывает SIGTERM как PID 1
→ 10 секунд ожидания → SIGKILL → некорректное завершение

с tini:
docker stop → SIGTERM → tini (PID 1) → node → корректное завершение
```

## 17.1. Load Average — средняя нагрузка

```bash
uptime
# 15:00:00 up 5 days, load average: 1.50, 0.75, 0.50
#                                    ↑     ↑     ↑
#                                   1мин  5мин  15мин
```

Load average — среднее количество процессов в очереди выполнения + ожидающих I/O за период.

### Как интерпретировать:
```
на 1-ядерной системе:
LA = 1.0 → ядро загружено на 100%, очереди нет
LA = 2.0 → ядро загружено на 100%, ещё столько же ждут
LA = 0.5 → ядро загружено на 50%

на 4-ядерной системе:
LA = 4.0 → все ядра загружены на 100%
LA = 2.0 → 50% загрузка
LA = 8.0 → система перегружена вдвое
```

```bash
nproc          # количество ядер CPU
lscpu          # подробная информация о CPU
```

Тренд важнее моментального значения:
```
LA 1мин > LA 15мин → нагрузка растёт
LA 1мин < LA 15мин → нагрузка спадает
```

<a id="q-17-2"></a>
## 17.2. Buffers vs Cache в памяти

```bash
free -h
#               total   used    free    shared  buff/cache  available
# Mem:          16G     4G      8G      100M    4G          11G
```

### Buffers:
```
буферы для блочных устройств (диск)
временное хранение данных при записи на диск
ядро накапливает записи и пишет блоками (эффективнее)
```

### Cache (page cache):
```
кэш содержимого файлов
файл прочитали → ядро держит в памяти
следующее чтение → из памяти (быстро)
```

**Важно:** `buff/cache` — не занятая память. Ядро освобождает её при нехватке памяти процессам. `available` — реально доступная память.

```bash
# освободить кaches (обычно не нужно — ядро сделает само):
echo 3 > /proc/sys/vm/drop_caches
```

<a id="q-17-4"></a>
## 17.4. iowait в метриках CPU

```bash
top
# %Cpu(s): 5.0 us, 2.0 sy, 0.0 ni, 83.0 id, 10.0 wa, 0.0 hi, 0.0 si
#                                              ↑
#                                           iowait
```

**iowait** — процент времени когда CPU был свободен но процессы ждали завершения I/O операций.

### Почему высокий iowait не всегда = проблема диска:
```
iowait означает: CPU мог бы работать, но ждал I/O
причины:
- медленный диск (реальная проблема)
- нормальное поведение (много операций чтения/записи)
- сеть (NFS, сетевые диски)
- swap thrashing

диагностика: смотреть вместе с iostat
высокий iowait + высокий await в iostat = проблема диска
высокий iowait + низкий await = много операций, диск справляется
```

## 17.6. Steal time в виртуализированной среде

```bash
top
# %Cpu(s): 5.0 us, 2.0 sy, 0.0 ni, 83.0 id, 0.0 wa, 0.0 hi, 0.0 si, 5.0 st
#                                                                         ↑
#                                                                       steal
```

**Steal time (st)** — время которое гипервизор "украл" у твоей VM для других VM на том же физическом хосте.

```
физический сервер: 8 ядер
VM1: 4 виртуальных ядра
VM2: 4 виртуальных ядра
VM3: 4 виртуальных ядра (overcommit)

гипервизор отдаёт ядра тому кто активен сейчас
VM1 хочет CPU → гипервизор занят VM2 → steal time у VM1
```

Высокий steal (>5-10%) = твой хостинг-провайдер перегрузил физический сервер. Нужно жаловаться или мигрировать.

## 17.8. Потери пакетов на уровне интерфейса

<a id="q-17-11"></a>
## 17.11. Диагностика утечки памяти (*) — USE методология

**USE методология** (Brendan Gregg) — для каждого ресурса проверить:
```
U — Utilization  (утилизация) — насколько занят ресурс
S — Saturation   (насыщение)  — есть ли очередь/ожидание
E — Errors       (ошибки)     — есть ли ошибки
```

Применение к сценарию "высокий LA, низкий CPU":
```bash
# CPU:
vmstat 1
# смотрим: r (run queue) > кол-во ядер → насыщение CPU
# смотрим: b (blocked) > 0 → процессы ждут I/O

# Диск:
iostat -x 1
# смотрим: await > 10ms → диск медленный
# смотрим: util > 80% → диск насыщен

# Память:
vmstat 1
# смотрим: si/so > 0 → swap активен → насыщение памяти
free -h
# смотрим: available < 10% → память почти закончилась

# Итог:
# высокий LA + низкий CPU + высокий iowait + swap = нехватка памяти → thrashing
```

### Диагностика скрытых потребителей памяти:
```bash
# slab allocator — память ядра:
cat /proc/slabinfo | sort -k3 -rn | head -20
slabtop   # интерактивно

# page cache:
cat /proc/meminfo | grep -E "Cached|Buffers|Slab"

# сетевые буферы сокетов:
ss -m   # память сокетов
cat /proc/net/sockstat
```
