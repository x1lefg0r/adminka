# Тема 2: Работа с файлами, ссылками и поиск

## 2.1. Inode

Inode — структура данных в файловой системе которая хранит **метаданные файла**, но не его имя и не содержимое.

### Что хранится в inode:
- Права доступа (`rwxr-xr-x`)
- Владелец (UID, GID)
- Размер файла
- Временные метки (atime, mtime, ctime)
- Указатели на блоки данных на диске
- Счётчик жёстких ссылок

### Что НЕ хранится:
- Имя файла — оно хранится в директории
- Содержимое — оно в блоках данных на диске

```bash
ls -i file.txt    # показать номер inode
stat file.txt     # все метаданные inode
stat -x file.txt  # на macOS
stat -c '%h' file.txt    # только счётчик ссылок
stat -c '%i' file.txt    # только номер inode
stat -c '%n %i %h' file.txt  # имя + inode + ссылки
```

### Блоки данных:
Диск разбит на блоки фиксированного размера — обычно 4096 байт (4KB). Inode хранит список номеров блоков, ядро читает их по порядку и получает данные.

```bash
ls -lh file.txt   # размер файла — реальные байты
du -h file.txt    # занято на диске — кратно размеру блока
```

### Правильная модель файла в Linux:

`touch file.txt` делает две вещи:
1. Создаёт inode (выделяет метаданные + блоки)
2. Создаёт запись в директории `"file.txt" → inode 12345`

**В Linux нет "файлов" в привычном смысле. Есть inode-ы и имена которые на них указывают.**

Директория — это просто таблица `имя → номер inode`.

```
"file.txt" ──→ inode 12345 ──→ [указатели на блоки] ──→ [блок 4521][блок 4522]
                                                          "hello world"
```

## 2.2. Жёсткие vs символические ссылки

```bash
ln file.txt hardlink      # жёсткая ссылка
ln -s file.txt symlink    # символическая ссылка
```

| | Жёсткая | Символическая |
|--|---------|---------------|
| Что это | Ещё одно имя для того же inode | Отдельный файл содержащий путь |
| Удалить оригинал | Данные живут пока счётчик > 0 | Ссылка становится битой |
| Между ФС | Нельзя (inode номера уникальны только внутри ФС) | Можно |
| На директорию | Нельзя (риск циклов) | Можно |

### Счётчик жёстких ссылок:
```bash
echo "hello" > file.txt   # счётчик = 1
ln file.txt hard1         # счётчик = 2
ln file.txt hard2         # счётчик = 3
rm file.txt               # счётчик = 2, данные живут
rm hard1                  # счётчик = 1, данные живут
rm hard2                  # счётчик = 0, данные удаляются с диска
```

**Правильное название удаления в Linux — `unlink`.** Ты не удаляешь данные — отвязываешь имя от inode. Данные исчезают когда счётчик = 0.

### Жёсткая ссылка — нет оригинала и копии:
```bash
ln file.txt hard
rm file.txt   # "оригинал" удалён
cat hard      # данные живы — hard такой же полноправный владелец
```

```
"file.txt" ─┐
             ├──→ inode 12345 ──→ данные на диске
"hard"     ─┘
```

Два указателя на одну структуру. Ни один не главнее.

## 2.3. Поиск пустых файлов и удаление

```bash
# сначала посмотреть
find /var/log -type f -empty

# потом удалить
find /var/log -type f -empty -delete
```

### Флаги `-type`:
```bash
-type f   # файлы
-type d   # директории
-type l   # символические ссылки
-type p   # именованные каналы (FIFO)
-type s   # сокеты
```

### Вместо `-empty` — фильтры по размеру:
```bash
-size 0           # пустые
-size +100M       # больше 100MB
-size -1k         # меньше 1KB
-size +1M -size -100M  # между 1MB и 100MB
```

По времени:
```bash
-mtime -1     # изменён меньше суток назад
-mtime +30    # изменён больше 30 дней назад
-newer file   # новее чем file
```

По правам:
```bash
-perm 644
-perm /111    # есть хоть один бит исполнения
```

### `-delete` vs pipe:
```bash
# безопасно с пробелами в именах:
find /var/log -type f -empty -print0 | xargs -0 rm
```

`-print0` разделяет нулевым байтом `\0` вместо `\n`. `-0` в xargs читает по `\0`. Нулевой байт не может быть в имени файла — безопасный разделитель.

### xargs:
Берёт stdin и передаёт как аргументы команде:
```bash
find /var/log -type f -empty | xargs rm
# превращает список файлов в: rm /var/log/a.log /var/log/b.log
```

`rm` не умеет читать stdin — только аргументы. xargs — мост между stdout одной команды и аргументами другой.

## 2.4. Временные метки файла

| Метка | Название | Когда обновляется |
|-------|----------|-------------------|
| `atime` | access time | при **чтении** файла |
| `mtime` | modify time | при изменении **содержимого** |
| `ctime` | change time | при изменении **метаданных** (права, владелец) или содержимого |

```bash
stat file.txt
# Access: 2026-06-24 ...   ← atime
# Modify: 2026-06-24 ...   ← mtime
# Change: 2026-06-24 ...   ← ctime
```

### Важные нюансы:
- `ctime` ≠ creation time (частое заблуждение). Время создания в Linux классически не хранится
- `mtime` всегда обновляет `ctime` — изменение содержимого меняет размер в inode (метаданные)
- Они расходятся когда меняешь метаданные без содержимого:

```bash
chmod 755 file.txt   # ctime обновился, mtime НЕ обновился
```

- `make`, `rsync`, бэкапы смотрят на `mtime` — "когда файл последний раз редактировался"

### `atime` часто отключают:
```bash
# в /etc/fstab:
/dev/sda1 / ext4 noatime 0 1
```
Каждое чтение файла вызывает запись на диск (обновление atime) — лишняя нагрузка.

## 2.5. Топ-10 самых больших файлов

```bash
# просто и надёжно:
du -ah /path | sort -rh | head -10
```

- `du` — disk usage
- `-a` — all, включить файлы (по умолчанию только директории)
- `-h` — human readable (KB/MB/GB)
- `sort -rh` — `-r` reverse, `-h` понимает суффиксы KB/MB/GB
- `head -10` — первые 10

```bash
# точнее — сортировка по байтам:
find /path -type f -printf '%s %p\n' | sort -rn | head -10 | awk '{print $2}' | xargs du -sh
```

- `-printf '%s %p\n'` — встроенный форматированный вывод find (не запускает внешний процесс)
  - `%s` — размер в байтах
  - `%p` — путь
- `sort -rn` — `-r` reverse, `-n` numeric (сортировать как числа, не строки)
- `awk '{print $2}'` — взять второй столбец (путь), отбросить размер в байтах

### awk:
**AWK** — Aho, Weinberger, Kernighan (фамилии авторов). Язык обработки текста.

```bash
awk '{print $2}'   # для каждой строки напечатать второе поле
#   ↑            ↑
#   одинарные кавычки bash — чтобы $2 не раскрылся как переменная bash
```

Внутри awk: `{}` — блок на каждую строку, `$1/$2/...` — поля разделённые пробелом.

## 2.6. Почему удаление файла не освобождает место

У inode два счётчика:
- **hardlink count** — количество имён в директориях
- **open file descriptor count** — количество процессов держащих файл открытым

Место освобождается только когда **оба счётчика = 0**.

```bash
# процесс пишет в nginx.log
rm nginx.log        # hardlink count = 0, но fd count = 1 (nginx держит открытым)
                    # место НЕ освободилось, nginx продолжает писать
df -h               # место всё ещё занято
```

### Найти такие файлы:
```bash
lsof | grep deleted   # открытые но удалённые файлы
```

### Освободить место без перезапуска процесса:
```bash
> /proc/PID/fd/3   # обнулить через файловый дескриптор процесса
```

### lsof — list open files:
Показывает все открытые файлы в системе прямо сейчас (файлы, сокеты, пайпы — в Linux всё файл).

```bash
lsof
# COMMAND  PID  USER  FD   TYPE  DEVICE  SIZE  NODE  NAME
# nginx   1234  root  3u   REG   sda1    512   4521  /var/log/nginx.log (deleted)
```

Столбцы:
- `FD` — номер дескриптора + режим (`r`=read, `w`=write, `u`=read+write)
- `TYPE` — `REG`=файл, `DIR`=директория, `IPv4`=сокет, `FIFO`=пайп, `CHR`=символьное устройство
- `NODE` — номер inode

```bash
lsof -p 1234     # -p = PID, все файлы процесса
lsof /var/log    # кто держит файлы в директории
lsof -i :80      # -i = internet, кто держит порт 80
lsof -u god      # -u = user, все файлы пользователя
lsof -i TCP      # все TCP соединения
lsof -i @1.2.3.4 # соединения с конкретным IP
```

### Файловые дескрипторы процесса:
```
0 → stdin
1 → stdout
2 → stderr
3, 4, 5... → файлы открытые в ходе работы (следующий свободный номер)
```

Каждый процесс имеет свою таблицу дескрипторов. Дескриптор `3` у процесса 1234 и `3` у процесса 5678 — разные, указывают на разные файлы.

### /proc/PID/fd/:
```bash
ls -la /proc/1234/fd/
# 0 → /dev/null
# 3 → /var/log/nginx.log (deleted)
```

Через дескриптор в `/proc` можно добраться до данных удалённого файла пока процесс его держит открытым (вопрос 2.12).

### Процесс:
Запущенная программа в памяти:
```
процесс = код (инструкции)
        + данные (переменные, heap)
        + стек (вызовы функций)
        + таблица дескрипторов (открытые файлы)
        + метаданные (PID, владелец, приоритет...)
```

Один бинарник можно запустить несколько раз — несколько независимых процессов с разными PID.

## 2.7. Битые символические ссылки в /etc

```bash
find /etc -type l ! -e   # -e = exists, ! = NOT
# или чище:
find /etc -xtype l       # -xtype = тип после разыменования
```

`-xtype l` — если ссылка разыменовалась успешно, тип станет `f` или `d`. Если цель не существует — тип остался `l` → битая ссылка.

## 2.8. Символическая ссылка на директорию

Практические сценарии:

**Версионирование:**
```bash
ln -s /opt/nginx-1.25 /opt/nginx   # все конфиги указывают на /opt/nginx
                                    # при обновлении меняешь только симлинк
```

**Унификация путей:**
```bash
ln -s /var/log/nginx /logs
```

**Совместимость:**
```bash
ln -s /usr/local/lib /usr/lib/mylib   # программа ожидает файлы в /usr/lib
```

Симлинк не работает внутри chroot/контейнера если путь цели там не существует. В таких случаях используют bind-mount (тема 7).

## 2.9. Копирование vs перемещение в пределах одной ФС

**`mv` в пределах одной ФС:**
- Создаёт новую запись в директории назначения
- Удаляет старую запись
- Inode не трогается, данные не копируются
- **Мгновенная операция** независимо от размера

**`cp`:**
- Создаёт новый inode
- Выделяет новые блоки
- Копирует данные побайтово
- Время зависит от размера

**`mv` между разными ФС:**
Вынужден делать `cp` + `rm` — уже не мгновенно.

### Что такое разные ФС:
```bash
df -h
# /dev/sda1   50G  /          # корневая ФС
# /dev/sda2   20G  /home      # отдельная ФС
# /dev/sdb1  500G  /data      # отдельный диск
# /dev/sdc1  100G  /var/log   # даже вложенная папка может быть отдельной ФС
```

Файловая система — раздел диска примонтированный в точку. Не обязательно на корневом уровне.

```bash
df /home/god/file.txt /data/   # проверить на каких ФС лежат пути
```

## 2.10. Что происходит с символической ссылкой при удалении цели

Ссылка становится **битой** — файл с путём который никуда не ведёт. Сама ссылка не удаляется автоматически.

```bash
echo "hello" > file.txt
ln -s file.txt symlink
rm file.txt

cat symlink    # → No such file or directory
ls -la symlink # → symlink -> file.txt  (ссылка есть, цели нет)
```

Жёсткая ссылка при удалении "оригинала" — данные живы:
```bash
ln file.txt hardlink
rm file.txt
cat hardlink   # → hello
```
